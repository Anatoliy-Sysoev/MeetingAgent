# Архитектура Генерации Итогов Встреч

Дата: 2026-05-08.

## Цель

`FTT-MA-12` должен давать не просто валидные файлы, а читаемые и проверяемые итоги встречи: memo, протокол, решения, задачи, риски и открытые вопросы.

Текущий `extractive`-режим в `scripts/07_generate_meeting_artifacts.py` считается скаффолдом контракта. Он полезен, чтобы проверить переход `transcribed -> summarized`, но не считается продуктовым генератором итогов: выбор сегментов по ключевым словам дает шумные решения и задачи без достаточного контекста.

## Production-Подход

Целевой путь генерации:

```text
transcript/segments.jsonl
        |
        v
[MAP] окна по 6 минут, overlap 30 секунд
        |
        v
_partials/window_<NN>.json
        |
        v
[REDUCE] объединение, дедупликация, нормализация
        |
        v
decisions.json / tasks.json / risks.json / open_questions.json
        |
        v
[RENDER] memo.md / protocol.md из финальных JSON
```

## MAP

MAP-этап получает одно окно transcript и извлекает только фактически подтвержденные элементы:

- темы обсуждения;
- решения;
- поручения;
- риски;
- открытые вопросы.

Окно по умолчанию:

- `--window-seconds 360`;
- `--window-overlap-seconds 30`;
- последнее окно короче 60 секунд сливается с предыдущим.

Временные ID должны содержать ID окна:

- `DEC-TMP-W03-001`;
- `TASK-TMP-W03-001`;
- `RISK-TMP-W03-001`;
- `Q-TMP-W03-001`;
- `TOPIC-TMP-W03-001`.

MAP-промпт: `configs/prompts/meeting_map_extract.md`.

## REDUCE

REDUCE-этап получает все partial artifacts и формирует финальный набор:

- `decisions.items`;
- `tasks.items`;
- `risks.items`;
- `open_questions.items`;
- `topics.items`.

Правила:

- временные ID не сохраняются;
- финальные ID перенумеровываются как `DEC-001`, `TASK-001`, `RISK-001`, `Q-001`, `TOPIC-001`;
- дубли объединяются только если совпадает объект и действие;
- вопросы, закрытые решением в другом окне, исключаются из `open_questions`;
- риск, снятый решением, переносится в темы и не остается активным риском;
- сомнительные пункты сохраняются с `needs_review = true`.

REDUCE-промпт: `configs/prompts/meeting_reduce_artifacts.md`.

## RENDER

RENDER-этап не должен становиться вторым источником истины.

Основное правило: `memo.md` и `protocol.md` строятся из финальных JSON-артефактов, а не из сырого transcript. Таблицы решений, задач, рисков и вопросов рендерятся детерминированно кодом. LLM можно использовать только для коротких текстовых разделов: краткая суть и ход обсуждения по темам.

RENDER-промпт для будущего LLM-слоя: `configs/prompts/meeting_render_documents.md`.

## Строгие Правила Классификации

- Не превращать вопросы в задачи.
- Не превращать риски в решения.
- Не превращать отрицание действия в поручение.
- Не считать решением короткие реакции без предмета: "да", "хорошо", "понял", "согласен".
- Каждый пункт должен быть понятен без чтения всего transcript.
- Каждый пункт должен иметь минимум один `source_ref` с таймкодом и короткой цитатой.
- Если пункт сомнительный, он остается в JSON с `needs_review = true`, а не удаляется молча.

## ASR-Нормализация

Промпты могут исправлять только очевидные ошибки распознавания:

- `рейсер`, `рейсере` -> `реестр`;
- `глазарий` -> `глоссарий`;
- `ферсинирование`, `кверсионирование` -> `версионирование`;
- `баке` -> `бакет`;
- `ЦИТА`, `CTA`, `СТА` -> `ЦТА`, если речь о целевой технической архитектуре;
- `FTT` -> `ФТТ`;
- `СЕМ`, `система 7`, `M7` -> `SIEM`, если речь о событиях ИБ или мониторинге ИБ.

Если исправление неочевидно, пункт получает `needs_review = true`.

## Промежуточные Артефакты

Промежуточные результаты являются runtime-данными и не коммитятся:

```text
meetings/<meeting_id>/_partials/window_<NN>.raw.txt
meetings/<meeting_id>/_partials/window_<NN>.json
meetings/<meeting_id>/_partials/reduce.raw.txt
meetings/<meeting_id>/_partials/reduce.json
```

Это нужно, чтобы можно было перезапустить только REDUCE или RENDER, если один из этапов упал или дал плохой Markdown.

## Модели И Параметры Для Эксперимента

Порядок проверки моделей:

1. `qwen2.5:7b-instruct`;
2. `mistral:7b-instruct`;
3. `qwen3:8b`, только если качество заметно выше и время приемлемо;
4. малые модели для MAP-окон, если качество классификации не проседает.

Стартовые параметры:

- `--temperature 0.1`;
- `--top-p 0.8`;
- `--num-ctx 8192`;
- `--timeout-sec 900`;
- `--window-seconds 360`;
- `--window-overlap-seconds 30`.

Перед полным прогоном нужно проверить одно окно transcript и измерить:

- время ответа;
- валидность JSON;
- качество классификации;
- долю пунктов с `needs_review = true`.

Если одно окно занимает больше 5 минут на CPU, модель или размер окна не подходят для текущего local-first профиля.

## Что Не Является Production-Путем

- Один большой prompt на весь transcript.
- Три независимых LLM-запроса: отдельно memo, отдельно protocol, отдельно JSON, потому что они могут рассинхронизироваться.
- Прямое копирование сырых фраз из transcript в решения или задачи.
- Keyword-only extractive как финальный генератор.
