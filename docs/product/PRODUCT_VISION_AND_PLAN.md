# Видение И Детальный План Продукта MeetingAgent

Обновлено: 2026-05-07.

## 1. Кратко

**MeetingAgent** - локальный агент памяти проекта. Он собирает проектные документы, записи встреч, транскрипты, протоколы, решения, задачи и требования в единую базу знаний, а затем помогает искать, объяснять, классифицировать и генерировать проектную документацию с опорой на источники.

Главная идея:

> Любая встреча и любой документ должны становиться частью управляемой памяти проекта, а не теряться в папках, чатах и видеоархиве.

Стратегическое видение остается в этом документе. Исполнительная карта этапов, ФТТ и критериев готовности ведется в `docs/product/PROJECT_STAGES_AND_FTT.md`.

## 2. Что Мы Реализуем

Мы строим не просто RAG и не просто транскрибатор. Мы строим продукт из пяти связанных возможностей:

1. **Память проекта**
   - индексирует проектные документы;
   - хранит chunks, embeddings и metadata;
   - позволяет задавать вопросы по проекту;
   - возвращает ответы с источниками.

2. **Обработка встреч**
   - принимает видео/аудио;
   - делает транскрипт;
   - выделяет суть, решения, риски, задачи, открытые вопросы;
   - формирует memo и протокол;
   - связывает встречу с этапом проекта, ФТТ, задачей и документами.

3. **Классификация и маршрутизация**
   - определяет, к какому этапу проекта относится материал;
   - связывает материал с ФТТ, проектным решением, архитектурой, задачей;
   - предлагает папку/документ назначения;
   - отправляет спорные случаи в ручной разбор.

4. **Генерация документов**
   - пользователь описывает, какой документ нужен;
   - агент находит источники в RAG;
   - формирует черновик;
   - показывает, на какие документы и встречи опирается текст.

5. **Операционный продукт**
   - мониторинг следит за долгими задачами;
   - сборка RAG продолжается после сбоев;
   - есть локальный API и позже UI;
   - все изменения продукта фиксируются в Git.

## 3. Для Кого

### Руководитель проекта

Нужно:

- быстро получать протоколы;
- видеть решения и задачи;
- понимать, что обсуждалось на встречах;
- не терять договоренности.

### Аналитик

Нужно:

- искать по ФТТ;
- собирать требования из разных документов;
- понимать, какие встречи влияют на требования;
- готовить выдержки и пояснения.

### Архитектор

Нужно:

- связывать решения с архитектурой;
- находить причины принятых решений;
- видеть связанные документы и встречи;
- готовить архитектурные разделы.

### Ответственный за сдачу

Нужно:

- собирать Паспорт ИС;
- готовить итоговую документацию;
- находить подтверждения по проектным материалам;
- проверять полноту и трассируемость.

## 4. Основные Объекты Продукта

### Проект

Корневая сущность. Содержит документы, встречи, этапы, ФТТ, задачи, решения и настройки.

### Источник

Любой исходный материал:

- `.docx`;
- `.pdf`;
- `.xlsx`;
- `.pptx`;
- `.txt`;
- `.md`;
- транскрипт;
- протокол;
- memo;
- видео/аудио.

### Chunk

Фрагмент текста, который можно искать через embeddings. Chunk должен иметь:

- `chunk_id` для reuse embeddings cache;
- `db_id` как совместимое служебное поле для поискового индекса и анализа дублей;
- путь к источнику;
- тип источника;
- номер фрагмента;
- размер;
- ссылку на исходный документ.

### Встреча

Запись и все артефакты вокруг нее:

- исходный файл;
- транскрипт с таймкодами;
- memo;
- протокол;
- решения;
- задачи;
- риски;
- открытые вопросы;
- классификация;
- связанные источники.

### Документ

Проектный или сгенерированный документ:

- Паспорт ИС;
- ФТТ;
- проектное решение;
- архитектурная записка;
- протокол;
- чеклист сдачи.

### Решение

Зафиксированная договоренность:

- что решили;
- кто участвовал;
- когда;
- на каком основании;
- где источник.

### Задача

Действие, которое нужно выполнить:

- формулировка;
- ответственный, если известен;
- срок, если известен;
- источник;
- связь с этапом/ФТТ/документом.

### Brief

Описание того, какой документ нужно сгенерировать:

- цель;
- структура;
- обязательные разделы;
- источники;
- ограничения;
- формат результата.

## 5. Главные Пользовательские Сценарии

### Сценарий 1. Собрать RAG По Проекту

Пользователь указывает папку проекта. MeetingAgent:

1. сканирует файлы;
2. извлекает текст;
3. режет текст на chunks;
4. строит embeddings;
5. собирает локальный numpy-индекс;
6. пишет отчеты, что обработано и что пропущено.

Результат:

- можно задавать вопросы по проекту;
- есть ссылки на документы;
- сборка продолжается после сбоя.

### Сценарий 2. Спросить Проект

Пользователь задает вопрос:

> Что должно входить в Паспорт ИС?

MeetingAgent:

1. ищет релевантные chunks;
2. группирует источники;
3. формирует ответ;
4. показывает ссылки на документы;
5. отдельно показывает уверенность и пробелы.

Результат:

- ответ не просто "из головы";
- видно, какие документы использованы;
- можно перейти к источнику.

### Сценарий 3. Обработать Встречу

В папке появляется видео. MeetingAgent:

1. ждет завершения записи файла;
2. транскрибирует;
3. создает карточку встречи;
4. пишет memo;
5. пишет протокол;
6. извлекает решения и задачи;
7. классифицирует по этапу/ФТТ/документу;
8. добавляет результаты в RAG.

Результат:

- встреча не теряется;
- ее можно найти;
- задачи и решения попадают в память проекта.

### Сценарий 4. Сгенерировать Документ

Пользователь пишет:

> Собери черновик Паспорта ИС. Включи назначение, состав системы, архитектуру, интеграции, требования ФТТ и документы для сдачи.

MeetingAgent:

1. уточняет brief;
2. ищет источники;
3. собирает план документа;
4. генерирует черновик;
5. добавляет ссылки на источники;
6. выделяет места, где не хватает данных.

Результат:

- появляется черновик;
- видно, откуда взяты утверждения;
- человек редактирует, а не начинает с нуля.

### Сценарий 5. Обновить Базу Знаний

Появились новые документы или изменились старые. MeetingAgent:

1. находит новые/измененные/удаленные файлы;
2. обновляет extracted text;
3. пересчитывает только нужные chunks;
4. переиспользует embeddings cache;
5. обновляет локальный поисковый индекс;
6. пишет отчет об изменениях.

Результат:

- RAG не устаревает;
- не нужно пересобирать все с нуля;
- удаленные или нерелевантные документы можно убрать.

## 6. Что Берем Из Speakr

Референс: <https://github.com/murtaza-nasir/speakr>

Speakr позиционируется как самостоятельно размещаемая платформа для AI-транскрибации и заметок. Он полезен нам не как кодовая база для копирования, а как продуктовый ориентир. У Speakr лицензия AGPL-3.0, поэтому код напрямую не переносим. Берем идеи и адаптируем под проектную документацию.

### Берем В Первую Очередь

#### 1. Карточка записи

В Speakr запись - это центральный объект. Нам нужна такая же логика, но проектная:

- исходное видео/аудио;
- транскрипт;
- memo;
- протокол;
- решения;
- задачи;
- связанные ФТТ;
- связанные документы;
- статус обработки.

#### 2. Синхронизация аудио и транскрипта

В Speakr есть audio-transcript sync: клик по тексту ведет к месту в аудио, текущий текст подсвечивается. Для нас это важно:

- проверять спорные места;
- быстро находить первоисточник решения;
- подтверждать протокол таймкодом.

#### 3. Режим "Спроси Все"

В Speakr есть Inquire Mode - семантический поиск по записям. У нас это станет режимом:

> Спроси проект

Он должен искать по:

- проектным документам;
- встречам;
- протоколам;
- memo;
- задачам;
- решениям;
- сгенерированным документам.

#### 4. Prompt-Теги

У Speakr есть smart tags с AI prompts и tag prompt stacking. Для MeetingAgent это одна из самых сильных идей.

Наши теги:

- `ФТТ`;
- `Паспорт ИС`;
- `Архитектура`;
- `Протокол`;
- `Решения`;
- `Задачи`;
- `Сдача`;
- `Интеграции`;
- `Риски`.

Пример stacking:

```text
Встреча + ФТТ + Паспорт ИС
```

Результат: агент извлекает из встречи только то, что важно для требований и Паспорта ИС.

#### 5. Автоэкспорт

Speakr умеет auto-export в инструменты знаний. Нам нужен экспорт в проектную структуру:

- transcript в папку встречи;
- protocol в папку протоколов;
- memo в папку memo;
- задачи в `tasks.md` или `tasks.xlsx`;
- выдержки для документов в папки соответствующих документов.

#### 6. Локальный API

У Speakr есть API для автоматизации. Нам нужен локальный API:

- `POST /rag/build`
- `POST /rag/update`
- `POST /query`
- `POST /meetings/process`
- `GET /meetings/{id}`
- `POST /documents/generate`
- `GET /status`

Это позволит подключать UI, планировщик, n8n/Make и другие инструменты.

#### 7. Retention И Protected Records

Speakr поддерживает retention и защиту записей от удаления. Для нас:

- архитектурные решения хранить всегда;
- встречи по сдаче хранить всегда;
- технические черновые созвоны можно архивировать;
- важные записи помечать `protected`.

#### 8. Идентификация Спикеров

У Speakr есть идентификация спикеров и голосовые профили при наличии WhisperX ASR-сервиса. Для нас это не MVP, но важная будущая функция:

- "Заказчик";
- "РП";
- "Архитектор";
- "Аналитик";
- "Разработчик".

Сначала делаем `Спикер 1/2/3`, позже - профили.

#### 9. Уроки Надежности Embeddings

В changelog Speakr есть важные продуктовые уроки:

- API embeddings должен иметь повторные попытки;
- re-embed не должен молча терять chunks;
- при ошибках старые chunks лучше сохранять до успешной замены;
- большие библиотеки требуют быстрый top-k search;
- stale chunks надо находить и переиндексировать.

Это напрямую применимо к нашему RAG lifecycle.

### Не Берем На Старте

- Полная многопользовательская админка.
- Публичный доступ к материалам.
- SSO/OIDC.
- Сложную систему тем.
- Полную копию UI.
- Копирование кода Speakr.

Эти вещи можно рассмотреть позже, когда локальный одиночный продукт станет стабильным.

## 8. Этапы Развития

## Этап 0. Стабилизировать Текущий RAG

Статус: базово готово, нужен baseline качества.

Что делаем:

- проверить `data/numpy_index/manifest.json`;
- убедиться, что embeddings cache покрывает текущие chunks;
- проверить smoke-запросы через `scripts/04_query.py`;
- зафиксировать известные проблемы.

Критерий готовности:

- есть done marker;
- `data/numpy_index/manifest.json` содержит ожидаемое число chunks;
- минимум 10 тестовых вопросов дают релевантные источники.

## Этап 1. Сделать Удобный "Спроси Проект"

Что делаем:

- улучшаем `scripts/04_query.py`;
- добавляем ссылки на источники;
- показываем top sources;
- добавляем режим raw/debug;
- добавляем evaluation-вопросы.

Критерий готовности:

- пользователь задает вопрос и получает ответ + источники;
- видно, какие документы использованы;
- можно быстро проверить релевантность.

## Этап 2. Инкрементальное Обновление RAG

Что делаем:

- `update_rag.ps1`;
- сравнение manifest;
- обработка новых/измененных/удаленных файлов;
- stale chunks;
- безопасная перестройка локального numpy-индекса;
- отчет обновления.

Критерий готовности:

- добавление нового документа не требует полной пересборки;
- удаленный документ исчезает из поиска;
- embeddings cache не теряется.

## Этап 3. Карточка Встречи

Что делаем:

- watcher входящей папки;
- транскрибация видео/аудио;
- сохранение transcript;
- генерация memo;
- генерация protocol;
- extraction решений/задач/рисков;
- папка карточки встречи.

Критерий готовности:

- новое видео превращается в набор markdown/docx-артефактов;
- карточку встречи можно открыть и понять встречу без просмотра всего видео;
- артефакты добавляются в RAG.

## Этап 4. Классификация С Учетом Проекта

Что делаем:

- классификатор этапа проекта;
- классификатор ФТТ;
- классификатор документа/задачи;
- confidence score;
- ручной разбор низкой уверенности;
- сохранение metadata маршрутизации.

Критерий готовности:

- встреча автоматически получает вероятный этап/ФТТ/документ;
- спорные случаи не портят базу, а уходят в ручной inbox.

## Этап 5. Генерация Документов

Что делаем:

- brief-шаблоны;
- генератор Паспорта ИС;
- генератор выдержек ФТТ;
- генератор архитектурной записки;
- генератор протокола;
- citations;
- export Markdown/DOCX.

Критерий готовности:

- пользователь описывает документ и получает черновик;
- каждый важный тезис имеет source reference;
- документ можно редактировать, а не писать заново.

## Этап 6. Локальный API

Что делаем:

- сервис FastAPI;
- endpoints для RAG, встреч, документов, status;
- job queue;
- logs/status;
- документация OpenAPI.

Критерий готовности:

- pipeline можно запускать не только из PowerShell;
- UI и автоматизации могут обращаться к локальному сервису.

## Этап 7. Локальный UI

Что делаем:

- dashboard;
- inbox;
- карточка встречи;
- поиск;
- генерация документов;
- ручной разбор;
- настройки.

Критерий готовности:

- пользователь может работать без терминала для основных действий.

## Этап 8. Качество, Управление, Retention

Что делаем:

- evaluation dashboard;
- protected records;
- retention policies;
- backup/export;
- audit log;
- правила удаления/архивации.

Критерий готовности:

## 9. MVP

MVP должен включать:

1. RAG по проектным документам.
2. Query с источниками.
3. Watchdog и продолжение после сбоев.
4. Обработка одной встречи из файла.
5. Transcript + memo + protocol + tasks.
6. Простая классификация встречи.
7. Добавление результатов встречи в RAG.
8. Генерация одного документа по brief.

MVP не обязан включать:

- красивый UI;
- многопользовательский режим;
- SSO;
- public sharing;
- voice profiles;
- сложную систему прав.

## 10. Критерии Готовности

Фича считается готовой, если:

- есть понятный пользовательский сценарий;
- есть способ запустить через CLI/API;
- есть лог/статус;
- есть обработка ошибок;
- локальные рабочие данные не коммитятся;
- документация обновлена;
- `docs/context.md` и `docs/todo.md` актуальны;
- изменения закоммичены и запушены.

## 12. Главный Принцип

MeetingAgent должен быть не "магическим чатиком", а инженерной системой памяти проекта:

- все важно сохраняется;
- все найденное можно проверить;
- каждый вывод имеет источник;
- каждая встреча становится частью проекта;
- каждый документ собирается из доказуемых материалов.
