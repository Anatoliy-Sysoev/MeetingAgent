# Таксономия Проекта И Типы Документов

Обновлено: 2026-05-07.

Этот документ задает единый реестр кодов, чтобы MeetingAgent одинаково классифицировал документы, встречи, задачи и создаваемые артефакты.

## Этапы Проектной Документации

| Код | Этап | Примеры Материалов |
| --- | --- | --- |
| `PRJ-00` | Inbox / не определено | Новые записи, неразобранные документы, спорные классификации. |
| `PRJ-01` | Обследование и исходные данные | Интервью, вводные заказчика, текущие процессы, ограничения. |
| `PRJ-02` | ФТТ и требования | Функциональные требования, технические требования, требования ИБ. |
| `PRJ-03` | Проектное решение и архитектура | Архитектурные схемы, проектные решения, интеграционные решения. |
| `PRJ-04` | Реализация и настройки | Маппинги, справочники, настройки, интеграции, рабочие задачи. |
| `PRJ-05` | Испытания и ПМИ | ПМИ, тестовые сценарии, дефекты, результаты проверок. |
| `PRJ-06` | Сдачная документация | Паспорт ИС, инструкции, регламенты, протоколы сдачи. |
| `PRJ-07` | Эксплуатация | Поддержка, мониторинг, доступы, резервное копирование. |
| `PRJ-08` | Управление проектом | Решения, риски, поручения, сроки, ответственные. |

## Типы Документов

| Код | Тип | Где Используется |
| --- | --- | --- |
| `DOC-FTT` | ФТТ проекта АСУ | Поиск требований, классификация встреч, генерация выдержек. |
| `DOC-FTT-EXTRACT` | Выдержка ФТТ | Подготовка тематических требований по модулю или задаче. |
| `DOC-PASSPORT-IS` | Паспорт ИС | Сдачная документация и генерация черновиков. |
| `DOC-PMI` | ПМИ | Испытания, приемка, проверка требований. |
| `DOC-PROJECT-SOLUTION` | Проектное решение | Архитектура, интеграции, техническая реализация. |
| `DOC-ARCH` | Архитектурная записка | Объяснение компонентов, потоков данных и решений. |
| `DOC-PROTOCOL` | Протокол встречи | Решения, поручения, вопросы, итоги совещаний. |
| `DOC-MEMO` | Memo встречи | Краткая суть встречи для быстрого поиска. |
| `DOC-TASKS` | Реестр задач | Поручения и действия из встреч или документов. |
| `DOC-REGULATION` | Регламент или инструкция | Эксплуатация, поддержка, рабочие процедуры. |
| `DOC-HANDOVER` | Комплект сдачи | Итоговая передача результата, чеклисты и акты. |

## Реестр ФТТ Проекта АСУ

Для `FTT-MA-13` нужен отдельный извлеченный реестр ФТТ проекта АСУ. Это не то же самое, что `FTT-MA-*`.

Минимальный формат будущего реестра:

```json
{
  "ftt_id": "ASU-FTT-001",
  "title": "Название требования",
  "text": "Формулировка требования",
  "source_path": "relative/path/to/source.docx",
  "chunk_id": "chunk-id",
  "project_stage": "PRJ-02",
  "document_type": "DOC-FTT",
  "confidence": 0.0,
  "notes": "что нужно проверить вручную"
}
```

## Как Извлекать ФТТ Проекта

1. Через RAG найти документы и фрагменты, похожие на требования.
2. Сформировать черновой реестр ФТТ с источниками.
3. Вручную проверить названия, формулировки и границы требований.
4. Использовать этот реестр как справочник для классификации встреч и задач.
5. При изменении ФТТ обновлять справочник и переиндексировать связанные артефакты.

## Правило Неуверенности

Если MeetingAgent не уверен в этапе, типе документа или пункте ФТТ, он должен:

- оставить материал в `PRJ-00`;
- показать 2-3 кандидата;
- объяснить причину выбора;
- указать источники;
- потребовать ручное подтверждение перед маршрутизацией.
