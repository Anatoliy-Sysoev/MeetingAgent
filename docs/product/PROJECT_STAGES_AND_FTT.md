# Этапы Проекта И ФТТ MeetingAgent

Обновлено: 2026-05-26.

Этот документ - рабочая карта реализации MeetingAgent. По нему можно идти как по чек-листу: какой этап делаем сейчас, какие пункты ФТТ закрываем, какой артефакт должен появиться и как понять, что этап готов.

## Термины

**ФТТ MeetingAgent** - функционально-технические требования к самому продукту MeetingAgent. В этом документе они обозначены как `FTT-MA-XX`.

**ФТТ проекта АСУ** - требования из реальной проектной документации заказчика. Их нельзя придумывать вручную: они должны извлекаться из RAG-источников и сохраняться с ссылками на документы.

**Этап продукта** - этап разработки MeetingAgent.

**Этап проектной документации** - предметная категория, к которой MeetingAgent должен привязывать встречу, документ, задачу или фрагмент: ФТТ, проектное решение, ПМИ, Паспорт ИС, сдача и так далее.

## Как Мы Идем По Карте

1. Сначала закрываем базу: устойчивый RAG, evaluation, понятные источники.
2. Потом делаем карточку встречи: видео, транскрипт, memo, протокол, решения, задачи.
3. Затем добавляем live-транскрибацию встречи как отдельный поток, чтобы заметки появлялись еще во время разговора.
4. После этого включаем классификацию: этап проекта, ФТТ, документ, задача, решение, риск.
5. Перед большим UI делаем узкий `Project-Only Chatbot MVP`: чат только по проектным источникам, с отказом без источников.
6. Только потом строим генерацию документов, расширенный local API и полноценный UI.

Такой порядок важен: генерация документов без проверенного RAG и трассировки источников будет выглядеть умно, но ей нельзя будет доверять.

## Нефункциональные Требования MVP

| Код | Требование | Целевой Ориентир |
| --- | --- | --- |
| `NFR-01` | Режим работы | Один пользователь на одной машине. Переход к 2-3 пользователям - отдельный этап через локальный сервис или общее хранилище. |
| `NFR-02` | Объем корпуса | До 20-30 тыс. chunks в горизонте 3-6 месяцев без обязательного FAISS. |
| `NFR-03` | Встречи | 2-3 встречи в день по 30-60 минут. |
| `NFR-04` | Offline-транскрибация | 60 минут аудио должны обрабатываться за приемлемое время на CPU; целевой ориентир - до 30-45 минут для MVP. |
| `NFR-05` | Live-транскрибация | Задержка несколько секунд допустима; точность важнее мгновенности. |
| `NFR-06` | Медиа | Нужно контролировать рост папки записей; ориентир лимита для MVP - 100 ГБ локальных медиа до архивации. |
| `NFR-07` | Восстановление | Потеря `data/embeddings_cache.jsonl` считается тяжелым инцидентом; нужен ежедневный backup. |

Подробные правила backup и хранения: `docs/operations/BACKUP_AND_RETENTION.md`.

## Предметные Этапы Проекта АСУ

MeetingAgent должен уметь относить документы, встречи и задачи к этим этапам. Это не этапы разработки продукта, а рабочая таксономия проектной документации.

| Код | Этап | Что Относится К Этапу |
| --- | --- | --- |
| `PRJ-00` | Inbox / не определено | Новые записи, документы и заметки, которые еще не классифицированы. |
| `PRJ-01` | Обследование и исходные данные | Интервью, текущие процессы, источники данных, ограничения, вводные заказчика. |
| `PRJ-02` | ФТТ и требования | Функциональные требования, технические требования, требования ИБ, интеграции, ограничения. |
| `PRJ-03` | Проектное решение и архитектура | Архитектура, компоненты, потоки данных, ЦТА, интеграционные схемы, проектные решения. |
| `PRJ-04` | Реализация и настройки | Разработка, настройка, маппинги, справочники, интеграции, миграции, рабочие задачи. |
| `PRJ-05` | Испытания и ПМИ | Программы и методики испытаний, сценарии проверки, дефекты, результаты тестов. |
| `PRJ-06` | Сдачная документация | Паспорт ИС, инструкции, регламенты, протоколы сдачи, комплекты документов. |
| `PRJ-07` | Эксплуатация | Поддержка, мониторинг, доступы, резервное копирование, регламентные операции. |
| `PRJ-08` | Управление проектом | Решения, риски, поручения, сроки, ответственные, протоколы совещаний. |

Минимальное правило: если MeetingAgent не уверен, он оставляет материал в `PRJ-00` и предлагает 2-3 кандидата с объяснением и ссылками на источники.

Единый реестр этапов и типов документов ведется в `docs/product/PROJECT_TAXONOMY.md`.

## ФТТ MeetingAgent

### `FTT-MA-01` Репозиторий И Память Проекта

**Требование:** каждый значимый шаг продукта должен быть закреплен в Git и проектных документах.

**Что уже есть:** `README.md`, `AGENTS.md`, `docs/context.md`, `docs/decisions.md`, `docs/todo.md`, `.env.example`, приватный GitHub repo.

**Критерий готовности:** новый тред может восстановить контекст по файлам проекта и `git log --oneline -10`.

**Статус:** готово.

### `FTT-MA-02` Инвентаризация Документов

**Требование:** система должна находить документы проекта в заданной папке и строить manifest.

**Артефакты:** `scripts/01_inventory.py`, manifest в `data/`.

**Критерий готовности:** видны количество файлов, типы, размеры, пропуски и ошибки.

**Статус:** работает.

### `FTT-MA-03` Извлечение Текста

**Требование:** система должна извлекать текст из поддерживаемых форматов: DOCX, PDF, PPTX, XLSX, HTML, TXT/MD и других рабочих файлов.

**Артефакты:** `scripts/02_extract_text.py`, extracted text в `data/`.

**Критерий готовности:** для каждого файла есть статус: извлечен, пропущен или ошибка с причиной.

**Статус:** работает, есть edge cases по нестандартным spreadsheets.

### `FTT-MA-04` Chunks И Embeddings

**Требование:** система должна резать текст на chunks, считать embeddings и продолжать работу после сбоев.

**Артефакты:** `data/chunks.jsonl`, `data/embeddings_cache.jsonl`.

**Параметры MVP:** `chunk_size_chars=3000`, `chunk_overlap_chars=300`. При изменении этих параметров часть cache становится устаревшей по дизайну, потому что меняются границы chunks.

**Стратегия:** chunks пока режутся по символам с попыткой разрыва на абзаце или предложении. Семантическая нарезка может появиться позже, если evaluation покажет проблему.

**Критерий готовности:** все актуальные chunks имеют embedding `bge-m3`, cache не пересчитывается без необходимости.

**Статус:** работает.

### `FTT-MA-05` Стабильный Локальный RAG Backend

**Требование:** поиск должен работать локально и не зависеть от нестабильного ChromaDB.

**Артефакты:** `scripts/rag_numpy_backend.py`, `scripts/05_build_numpy_index.py`, `data/numpy_index`.

**Критерий готовности:** JSON-запросы возвращают релевантные источники через numpy-поиск.

**Статус:** работает.

### `FTT-MA-06` Query С Источниками

**Требование:** RAG-запрос должен возвращать не только ответ, но и проверяемые источники: файл, chunk, score, фрагмент.

**Артефакты:** `scripts/04_query.py`.

**Критерий готовности:** пользователь может понять, из какого документа взят ответ.

**Статус:** работает. Для Project Knowledge Bot дополнительно реализованы `/search`, `/chat`, Web UI и Telegram adapter с выводом sources/citations.

### `FTT-MA-07` Evaluation И Baseline Качества

**Требование:** качество RAG должно проверяться набором контрольных вопросов.

**Артефакты:** `docs/quality/EVALUATION_PLAN.md`, `docs/quality/rag_eval_questions.md`, `docs/quality/rag_eval_baseline_clean_2026-05-07.md`.

**Критерий готовности:** есть 10-20 вопросов по ФТТ, ПМИ, Паспорту ИС, архитектуре и сдачной документации; для каждого зафиксированы ожидаемые источники и фактический top-k.

**Статус:** работает. Есть realistic-100/500, manual review pipeline, failure buckets, approved regression set, targeted bucket eval и smoke-наборы для NTK corpus. Продолжается улучшение по слабым buckets.

### `FTT-MA-08` Обработка Входящих Записей

**Требование:** система должна замечать новые видео/аудио в watched folder и создавать задачу обработки.

**Связь с обновлением RAG:** watcher создает job обработки встречи; после создания memo, протокола, решений, задач и финального transcript job запускает инкрементальное обновление RAG по этим артефактам.

**Артефакты:** `scripts/06_transcribe_meeting.py` для одной папки встречи; будущий watcher, очередь jobs.

**Критерий готовности:** новое медиа не теряется, не обрабатывается дважды и получает статус.

**Статус:** базовый CLI-слой готов. `scripts/20_ingest_meeting.py` принимает `mp4/mp3/wav/m4a`, создает `meetings/YYYY-MM-DD__slug/`, копирует исходное медиа в `source/` и пишет валидный `meeting.json`. Watcher и очередь jobs запланированы отдельно.

### `FTT-MA-09` Карточка Встречи

**Требование:** каждая встреча должна иметь единую папку и машинно-читаемую карточку.

**Минимальная структура:**

```text
meetings/
  YYYY-MM-DD__short-title/
    source/
    transcript/
    artifacts/
    exports/
    meeting.json
```

**Минимальные поля:** дата, название, участники, источник, статус обработки, этап проекта, связанные ФТТ, связанные документы, решения, задачи, риски.

**Что попадает в RAG:** memo, протокол, решения, задачи, риски, открытые вопросы и финальный transcript сегментами с metadata.

**Что не попадает в RAG:** исходное видео/аудио, временные файлы и сырой live transcript, если он еще не очищен.

**Формальный контракт:** JSON schema зафиксирована в `configs/schemas/meeting.schema.json`, человекочитаемый шаблон - в `docs/templates/MEETING_CARD.md`.

**Критерий готовности:** любую встречу можно открыть как самостоятельный пакет данных.

**Статус:** базовый слой готов. Есть `configs/schemas/meeting.schema.json`, `docs/templates/MEETING_CARD.md`, architecture/scope docs и `scripts/20_ingest_meeting.py`, который создает карточку по каноническому контракту. Следующий шаг - ASR wrapper и speaker transcript.

### `FTT-MA-10` Live-Транскрибация Встречи

**Требование:** MeetingAgent должен уметь писать черновой транскрипт во время встречи, а не только после появления готового видео.

**Источники аудио:**

- микрофон пользователя;
- системный звук через loopback;
- смешанный поток, если включены оба источника.

**Правило хранения:** `MIC` и `SYS` сохраняются как отдельные дорожки/сегменты. `MIX` считается производным потоком для просмотра и не должен быть единственным источником истины.

**CPU-first профиль для MVP:**

- faster-whisper;
- модель `small`;
- compute `int8`;
- язык `ru`;
- chunk 5 секунд;
- beam 3;
- VAD выключен по умолчанию для более предсказуемых live-таймкодов.

**Выходы live-сессии:**

- `live_transcript.txt` - читаемый черновик;
- `live_segments.jsonl` - сегменты с `source`, `start`, `end`, `text`, timestamp;
- `live_subtitles.srt` - опционально, экспериментально;
- `session.json` - параметры сессии и источники аудио.

**Критерий готовности:** можно нажать старт, получить обновляемый transcript во время встречи, остановить сессию и передать ее в обычный pipeline memo/protocol/classification.

**Статус:** запланировано, есть экспериментальный референс WhisperDesk.

### `FTT-MA-11` Источники Аудио И Таймкоды

**Требование:** система должна сохранять источник аудио и таймкоды каждого сегмента.

**Почему важно:** без источника и таймкодов сложно отличить речь пользователя от речи собеседника, вернуться к месту в записи и делать проверяемый протокол.

**Критерий готовности:** каждый сегмент transcript имеет `source`, `start`, `end`, `text`; для live MVP source должен различать минимум `MIC`, `SYS` и `MIX`.

**Статус:** частично готово для offline-записей: `scripts/21_extract_audio.py` создает `source/audio_16k_mono.wav` в формате mono 16000 Hz, добавляет его в `source.media_files` и помечает как no-index artifact. Сегменты transcript имеют `start/end/text`; live-источники `MIC/SYS/MIX` и diarization пока запланированы. Для MVP meeting pipeline допустим `SPEAKER_UNKNOWN`.

### `FTT-MA-12` Memo, Протокол, Решения, Риски И Задачи

**Требование:** после транскрибации MeetingAgent должен создавать структурированные итоги.

**Выходы:**

- memo: краткая суть встречи;
- протокол: вопросы, обсуждение, решения;
- action items: задачи, ответственные, сроки;
- risks: риски и открытые вопросы;
- decisions: принятые решения с источниками.

**Артефакты MVP:**

- `configs/prompts/meeting_memo.md`;
- `configs/prompts/meeting_protocol.md`;
- `configs/prompts/meeting_artifacts_json.md`;
- `configs/prompts/meeting_map_extract.md`;
- `configs/prompts/meeting_reduce_artifacts.md`;
- `configs/prompts/meeting_render_documents.md`;
- `configs/schemas/meeting.decisions.schema.json`;
- `configs/schemas/meeting.tasks.schema.json`;
- `configs/schemas/meeting.risks.schema.json`;
- `configs/schemas/meeting.open_questions.schema.json`;
- `scripts/07_generate_meeting_artifacts.py`.

**Критерий готовности:** каждый пункт связан с transcript segment или RAG-источником.

**Статус:** начато. Extractive-режим прошел на тестовой встрече и создал валидные артефакты, но по качеству является только скаффолдом: он выбирает сегменты по ключевым словам и не считается готовым продуктовым генератором. Целевой production-путь зафиксирован как `MAP -> REDUCE -> RENDER` в `docs/architecture/MEETING_ARTIFACTS_PIPELINE.md`; `scripts/08_process_meeting_pipeline.py` прошел 2-window smoke и умеет сохранять partial JSON, resume и нормализованные `source_refs`. Следующий шаг - полный прогон и ручная оценка качества memo/protocol.

### `FTT-MA-13` Классификация По Этапу, ФТТ, Документу И Задаче

**Требование:** MeetingAgent должен определять, к чему относится встреча или фрагмент.

**Классификация:**

- этап проектной документации: `PRJ-00` - `PRJ-08`;
- пункт ФТТ проекта АСУ, если найден в источниках;
- документ или тип документа: Паспорт ИС, ПМИ, проектное решение, архитектура, регламент;
- задача или поручение;
- уверенность классификации и объяснение.

**Предусловие:** перед надежной классификацией по ФТТ нужно извлечь черновой реестр ФТТ проекта АСУ из источников RAG и вручную проверить его. Минимальный формат реестра описан в `docs/product/PROJECT_TAXONOMY.md`.

**Критерий готовности:** система показывает top candidates и не выдает догадку за факт.

**Статус:** запланировано.

### `FTT-MA-14` Генерация Документов На Основе Источников

**Требование:** MeetingAgent должен помогать составлять документы на основе уже имеющейся документации и встреч.

**Примеры:**

- Паспорт ИС из ФТТ, проектного решения, архитектуры и решений встреч;
- выдержка требований по конкретному модулю;
- архитектурная записка;
- протокол совещания;
- список открытых вопросов для заказчика.

**Критерий готовности:** черновик содержит структуру, ссылки на источники и список мест, требующих ручной проверки.

**Статус:** запланировано.

### `FTT-MA-15` Инкрементальное Обновление RAG

**Требование:** новые документы и созданные MeetingAgent артефакты должны добавляться в RAG без полной пересборки.

**Сценарии:**

- добавлен новый документ;
- изменен существующий документ;
- удален нерелевантный документ;
- создан Паспорт ИС или протокол, который тоже нужно включить в память проекта.

**Deletion handling:** incremental update должен явно удалять из актуальных `chunks` и индекса документы, которых больше нет в manifest или которые теперь попали под `exclude_path_patterns`. Stale embeddings могут оставаться в `embeddings_cache.jsonl`, но не должны попадать в `data/numpy_index`.

**Связь со встречами:** новые артефакты встречи добавляются в RAG после завершения обработки встречи, но исходные медиа не индексируются.

**Критерий готовности:** есть `update_rag`-команда с понятным отчетом: добавлено, обновлено, удалено, пропущено.

**Статус:** запланировано.

### `FTT-MA-16` Локальный API, Задачи И Статусы

**Требование:** продукт должен иметь локальный API для запуска задач и просмотра статусов.

**Минимальные endpoints:**

- health;
- search;
- ingest document;
- ingest meeting;
- job status;
- meeting artifacts;
- generate document draft.

**Критерий готовности:** UI и CLI используют один контракт локального сервиса.

**Статус:** запланировано.

### `FTT-MA-17` Локальный UI

**Требование:** пользователь должен видеть inbox, встречи, документы, поиск, генерацию и статусы без ручного запуска отдельных скриптов.

**Поверхности:**

- сначала простой локальный web-интерфейс;
- позже desktop wrapper, если потребуется.

**Критерий готовности:** можно обработать встречу и открыть результат из интерфейса.

**Статус:** запланировано.

### `FTT-MA-18` Безопасность И Local-First

**Требование:** проектные документы, транскрипты и индексы по умолчанию остаются локально.

**Правила:**

- не коммитить runtime-данные;
- не отправлять реальные проектные материалы в cloud без явного решения;
- `.env` и `config.yaml` не попадают в Git;
- cloud/Colab/Kaggle использовать только для синтетических, публичных или обезличенных данных.

**Критерий готовности:** репозиторий можно пушить без утечки документов и секретов.

**Статус:** частично готово.

### `FTT-MA-19` Эксплуатация, Watchdog И Восстановление

**Требование:** долгие задачи должны переживать сбои Ollama, зависания и перезапуски.

**Артефакты:** `run_full_rag.ps1`, `monitor_rag.ps1`, done/failed markers, lock, cache.

**Критерий готовности:** мониторинг не запускает второй build, не убивает живой worker, умеет восстановить Ollama, а backup-правила для `embeddings_cache.jsonl` описаны и проверены.

**Статус:** частично готово.

### `FTT-MA-20` Экспорт И Форматы Артефактов

**Требование:** результаты должны сохраняться в форматах, удобных человеку и машине.

**Форматы:**

- Markdown для черновиков и ревью;
- JSON/JSONL для машинной обработки;
- DOCX для сдачных документов;
- SRT/VTT для субтитров, если нужны;
- TXT для простого transcript.

**Критерий готовности:** один и тот же результат можно читать, индексировать и экспортировать.

**Статус:** запланировано.

### `FTT-MA-21` Project-Only Chatbot MVP

**Требование:** MeetingAgent должен иметь очищенный чат-бот, который отвечает только на вопросы, связанные с проектом, и только при наличии проверяемых источников в RAG.

**Связанные требования:** `FTT-MA-05`, `FTT-MA-06`, `FTT-MA-07`, `FTT-MA-16`, `FTT-MA-17`, `FTT-MA-18`.

**Минимальное поведение:**

- если вопрос связан с проектной документацией, бот ищет источники в `data/numpy_index`;
- если источники найдены выше порога качества, бот отвечает на русском языке и показывает источники;
- если источников нет, score низкий или вопрос вне проекта, бот отказывает и не использует общие знания модели;
- каждый ответ содержит проверяемые `sources`: файл, chunk, score и короткий фрагмент;
- системные инструкции, секреты, `.env`, `config.yaml` и runtime-данные не раскрываются.

**Артефакты MVP:**

- `docs/product/PROJECT_ONLY_CHATBOT_MVP.md`;
- project-only prompt/policy;
- CLI и локальный API endpoint `/chat`;
- evaluation-набор вопросов для чат-бота;
- отчет первого smoke-прогона.

**Критерий готовности:** пользователь может задать вопрос по проекту, получить ответ с источниками или корректный отказ. Внепроектные вопросы и вопросы без источников не приводят к выдуманным ответам.

**Статус:** реализовано для Project Knowledge Bot: `POST /search`, `POST /chat`, `GET /ui`, локальный Web UI, Telegram adapter и QH-1..QH-5 release stabilization. Следующий этап - Docker и перенос lessons learned в общий MeetingAgent UI/jobs слой.

## Ближайший Практический Маршрут

### Сессия 1: Зафиксировать Качество RAG

Статус: выполнено и расширено.

Цель: понять, насколько текущий numpy/hybrid-поиск пригоден для реальных задач.

Делаем:

- берем 10-20 вопросов по ФТТ, ПМИ, Паспорту ИС, архитектуре и сдаче;
- запускаем `scripts/04_query.py --raw`;
- фиксируем top-k источники, дубли, провалы, ожидаемые документы;
- решаем, нужен ли reranking или чистка backup-документов.

Закрывает: `FTT-MA-07`. Сейчас продолжается bucket-driven улучшение retrieval quality.

### Сессия 2: Спроектировать Карточку Встречи

Статус: выполнено как контракт, требуется рабочий ingest.

Цель: договориться о формате, в который попадут и offline-записи, и live-сессии.

Делаем:

- JSON schema `meeting.json`;
- markdown-шаблон карточки;
- папочная структура;
- статусы обработки;
- список обязательных артефактов.

Закрывает: `FTT-MA-09`, `FTT-MA-20`. Следующий шаг: `20_ingest_meeting.py` и `21_extract_audio.py`.

### Сессия 3: Offline Meeting MVP

Цель: пройти один реальный файл через каноническую карточку встречи.

Делаем:

- `20_ingest_meeting.py`;
- `21_extract_audio.py`;
- ASR через `06_transcribe_meeting.py` или thin wrapper `22_transcribe_meeting.py`;
- `speaker_transcript.jsonl` с `SPEAKER_UNKNOWN`;
- meeting-aware chunks;
- indexing/search по meeting chunks.

Закрывает практический срез `FTT-MA-08`, `FTT-MA-09`, `FTT-MA-11`, частично `FTT-MA-12`, `FTT-MA-15`.

### Сессия 4: Live-Транскрибация

Цель: вынести из WhisperDesk не код как есть, а рабочий опыт.

Делаем:

- отдельный модуль audio capture;
- отдельный модуль live segmentation;
- отдельный writer TXT/JSONL/SRT;
- настройки CPU-first;
- интеграцию live-сессии с карточкой встречи.

Закрывает: `FTT-MA-10`, `FTT-MA-11`.

### Сессия 5: Memo И Протокол

Цель: превратить transcript в полезный проектный артефакт.

Делаем:

- prompt memo;
- prompt протокола;
- schema решений, рисков и задач;
- ссылки на transcript segments;
- RAG-подтверждение по документам.

Закрывает скаффолд `FTT-MA-12`; production-готовность требует map-reduce-render прогона и ручной оценки качества.

### Сессия 6: Классификация По Этапам И ФТТ

Цель: MeetingAgent должен сказать, к какому этапу и пунктам ФТТ относится встреча.

Делаем:

- prompt/classifier для `PRJ-*`;
- retrieval по ФТТ проекта АСУ;
- top candidates с confidence;
- ручная корректировка классификации;
- запись результата в `meeting.json`.

Закрывает: `FTT-MA-13`.

### Сессия 7: Project-Only Chatbot MVP

Статус: реализовано, следующий шаг Docker.

Цель: сделать первый очищенный чат по проектной памяти.

Делаем:

- project-only policy: отвечать только по источникам проекта;
- минимальный CLI или local API `/chat`;
- поиск через текущий numpy-RAG;
- отказ при отсутствии источников или внепроектном вопросе;
- вывод источников: файл, chunk, score, preview;
- smoke-набор вопросов по ФТТ, ПМИ, Паспорту ИС, архитектуре и один внепроектный вопрос.

Закрывает первый рабочий срез `FTT-MA-21` и частично закрывает `FTT-MA-16`, `FTT-MA-17`, `FTT-MA-18`.

## Что Не Делаем Пока

- Не строим большой UI до появления стабильной карточки встречи; исключение - узкий `Project-Only Chatbot MVP` как первая полезная поверхность RAG.
- Не обучаем модели на реальных проектных данных в cloud.
- Не переносим монолитный WhisperDesk как продуктовый код.
- Не генерируем сдачные документы без evaluation и ссылок на источники.
