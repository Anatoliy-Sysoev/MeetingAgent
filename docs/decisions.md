# Решения

## 2026-05-06 - Одна Папка, Один Git-Репозиторий

Решение: каждый пет-проект получает отдельную папку и отдельный Git-репозиторий.

Почему:

- зависимости остаются изолированными;
- история остается читаемой;
- проекты можно независимо переносить, пушить, архивировать и удалять;
- Codex может восстановить контекст по файлам репозитория.

## 2026-05-06 - Обязательные Файлы Памяти Проекта

Решение: в каждом пет-проекте должны быть:

- `README.md`;
- `AGENTS.md`;
- `docs/context.md`;
- `docs/decisions.md`;
- `docs/todo.md`;
- `.env.example`.

Почему:

- новые треды быстрее восстанавливают контекст;
- решения не теряются;
- следующие действия остаются явными;
- секреты и локальное состояние не попадают в Git.

## 2026-05-06 - Local-First По Умолчанию

Решение: MeetingAgent по умолчанию обрабатывает проектные документы локально.

Почему:

- проектные файлы и транскрипты встреч могут быть конфиденциальными;
- локальная Ollama и локальные Whisper-совместимые модели уменьшают риск утечки данных;
- cloud-интеграции можно добавить позже как явный opt-in.

## 2026-05-06 - Не Коммитить Локальные Рабочие Данные

Решение: игнорировать `.venv/`, `config.yaml`, `data/`, `logs/`, `vector_db/`, `watched_folder/` и медиафайлы.

Почему:

- эти файлы могут быть большими;
- они могут содержать конфиденциальные документы или транскрипты;
- это machine-specific runtime state, а не исходники продукта.

## 2026-05-06 - bge-m3 С num_ctx 8192

Решение: каждый Ollama embedding-запрос должен отправлять `options.num_ctx=8192` и `keep_alive=24h`.

Почему:

- Ollama может по умолчанию запускать `bge-m3` с контекстом 4096;
- реальные проектные chunks после токенизации могут превышать этот лимит;
- явный контекст предотвращает HTTP 500 из-за context overflow.

## 2026-05-06 - Сохранять Embedding Cache

Решение: оставлять `embedding_model="bge-m3"` в cache records и не удалять валидные cached embeddings при восстановлении.

Почему:

- cached embeddings получены теми же весами модели;
- `num_ctx` меняет допустимую длину входа, а не векторы для входов, которые уже помещались;
- сохранение cache делает многодневную сборку продолжаемой.

## 2026-05-06 - Разделить chunk_id И db_id

Решение: использовать `chunk_id` для переиспользования embedding cache, а `db_id` для уникальной идентичности записей в ChromaDB.

Почему:

- одинаковые backup-документы могут давать одинаковый `chunk_id`;
- ChromaDB требует уникальные IDs;
- добавление relative path в `db_id` устраняет collisions без инвалидирования существующего embeddings cache.

Уточнение от 2026-05-07: основной build больше не пишет в ChromaDB, но `db_id` остается в chunks как совместимое служебное поле на случай будущей миграции или анализа дублей.

## 2026-05-06 - Мониторинг Перезапускает Ollama, А Не Python

Решение: при зависании роста cache мониторинг перезапускает Ollama, но не убивает живой `03_build_index.py`.

Почему:

- Python worker имеет retry/backoff и может восстановиться после возвращения Ollama;
- убийство процесса создает лишний риск прерывания;
- удалять lock безопасно только при отсутствии живого build worker.

## 2026-05-06 - Русский Язык Документации

Решение: основная документация MeetingAgent ведется на русском языке.

Почему:

- продукт создается под русскоязычный рабочий процесс;
- проектные термины вроде ФТТ, Паспорт ИС, протокол и сдачная документация должны быть естественными;
- новый тред Codex быстрее восстанавливает контекст без перевода терминов.

## 2026-05-06 - Speakr Как Продуктовый Референс

Решение: использовать Speakr как источник продуктовых паттернов, но не копировать код.

Почему:

- Speakr хорошо показывает UX вокруг записей, транскриптов, поиска, prompt-тегов, API и retention;
- лицензия Speakr AGPL-3.0 требует аккуратности, поэтому берем идеи, а не реализацию;
- MeetingAgent решает другой домен: проектная память, ФТТ, протоколы, сдачная документация и трассируемость.

## 2026-05-07 - Numpy-Поиск Вместо Chroma В Критическом Пути

Решение: вынести поиск в отдельный локальный numpy-индекс, который строится из `data/chunks.jsonl` и `data/embeddings_cache.jsonl`, а ChromaDB больше не использовать как обязательную рабочую зависимость для запросов.

Почему:

- локальная ChromaDB может ломаться на загрузке HNSW-индекса;
- embeddings cache уже содержит валидные `bge-m3` векторы и позволяет собрать индекс без повторных запросов к Ollama;
- numpy-поиск проще диагностировать, атомарно пересобирать и переносить;
- FAISS можно подключить позже как ускорение, не меняя контракт поиска.

Уточнение от 2026-05-07: `scripts/03_build_index.py` больше не пересоздает `vector_db/`. Полная сборка RAG теперь строит chunks и embeddings cache, а затем `scripts/05_build_numpy_index.py` собирает основной поисковый индекс.

## 2026-05-07 - vector_db Не Отслеживается Git

Решение: полностью исключить `vector_db/` из Git, включая `.gitkeep` и README-заглушки.

Почему:

- `vector_db/` содержит локальные рабочие артефакты ChromaDB;
- старые версии `03_build_index.py` могли полностью пересоздавать эту папку;
- отслеживаемые Git-заглушки удалялись во время прошлых пересборок и создавали грязный `git status`;
- основной поиск теперь работает через `data/numpy_index`, поэтому `vector_db/` остается устаревшей локальной папкой.

## 2026-05-07 - Мониторинг Использует PID Из Lock-Файла

Решение: `monitor_rag.ps1` должен уметь определять живой build не только по `CommandLine` через CIM, но и по PID из `logs/build_index.lock`.

Почему:

- на Windows чтение `CommandLine` через CIM может быть нестабильным или недоступным;
- если lock содержит живой PID, мониторинг не должен удалять lock и запускать второй build;
- запасная проверка снижает риск дублей сборки и случайного вмешательства в долгий процесс.

## 2026-05-07 - ФТТ MeetingAgent Как Execution Checklist

Решение: вести отдельную карту этапов и ФТТ продукта в `docs/product/PROJECT_STAGES_AND_FTT.md`.

Почему:

- продукт уже вышел за рамки одной RAG-сборки и требует управляемой последовательности работ;
- для каждой функции нужен понятный статус, артефакт и критерий готовности;
- MeetingAgent должен различать ФТТ самого продукта и ФТТ проекта АСУ из пользовательской документации;
- карта этапов позволит двигаться от RAG к встречам, live-транскрибации, классификации и генерации документов без потери фокуса.

## 2026-05-07 - WhisperDesk Только Как Экспериментальный Референс

Решение: использовать `C:\Users\Сотрудник\Desktop\AI\WhisperDesk` как источник проверенных идей для live-транскрибации, но не переносить его монолитный код как продуктовую основу MeetingAgent.

Почему:

- эксперимент уже показывает полезные решения: mic + system audio loopback, CPU-first профиль `small/int8`, chunk 5 секунд, TXT/JSONL/SRT и `session.json`;
- MeetingAgent требует модульной архитектуры: audio capture, live segmentation, writers, карточка встречи и дальнейший pipeline memo/protocol/classification;
- перенос монолитного Tkinter-приложения усложнит поддержку и смешает эксперимент с продуктовым кодом.

## 2026-05-07 - MVP Рассчитан На Одного Пользователя

Решение: ближайший MVP MeetingAgent считается однопользовательским локальным продуктом.

Почему:

- текущие lock-файлы, `data/` и `logs/` рассчитаны на одну машину и один активный pipeline;
- в ближайшие месяцы важнее стабилизировать RAG, встречи, классификацию и генерацию;
- переход на 2-3 пользователей требует отдельного архитектурного решения: локальный сервис с общей папкой данных или сетевое хранилище с аккуратной блокировкой.

## 2026-05-07 - Live-Транскрибация Хранит Раздельные Дорожки

Решение: для live MVP основные данные должны храниться как отдельные дорожки `MIC` и `SYS`, а `MIX` считается производным потоком для просмотра.

Почему:

- если сохранить только смешанный поток, потом сложнее добавить diarization и проверку источника речи;
- раздельные дорожки позволяют понять, что сказал пользователь, а что пришло из системного звука;
- `MIX` полезен для быстрого чтения, но не должен быть единственным источником истины.

## 2026-05-07 - В RAG Индексируются Не Все Артефакты Встречи

Решение: исходные медиа не индексируются; в RAG попадают memo, протокол, решения, задачи, риски, открытые вопросы и финальный transcript с metadata.

Почему:

- сырой live transcript может быть шумным и ухудшать поиск;
- структурированные артефакты дают более качественные источники для классификации и генерации документов;
- transcript все равно нужен для проверки таймкодов и контекста, но его нужно индексировать сегментами, а не огромным файлом целиком.

## 2026-05-07 - Контракт Вывода `04_query.py`

Решение: `scripts/04_query.py` поддерживает три режима вывода - LLM-ответ по умолчанию, `--compact` для человека и `--raw` в JSON для evaluation и автоматизаций. В `--compact` и `--raw` для каждого источника выводятся `score` в диапазоне `0..1`, `relative_path`, `chunk_index` и preview-фрагмент.

Почему:

- `distance = 1 - cosine` интуитивно хуже читается и мешает быстро сравнивать результаты оценки;
- компактный текстовый вывод нужен для ручной проверки;
- JSON-режим нужен для исторических baseline-отчетов, regression-наборов и автоматизаций;
- режим LLM-ответа должен оставаться отдельным, чтобы не смешивать качество поиска и качество генерации.

## 2026-05-07 - Политика Ротации Медиа Как Архитектурное Решение

Решение: записи встреч без демонстрации экрана архивируются или удаляются по таймауту согласно `docs/operations/BACKUP_AND_RETENTION.md`, а записи с демонстрацией экрана хранятся как `protected` и не удаляются автоматически. Транскрипты, memo, протоколы, решения и задачи сохраняются вне зависимости от судьбы исходного медиа.

Почему:

- при потоке в несколько встреч в день бесконтрольное хранение медиа быстро становится неприемлемым;
- долговременная ценность встречи живет в артефактах, а не в самом видео;
- демонстрация экрана может содержать уникальные проектные данные, которых нет в документах, поэтому такие записи защищаются отдельно;
- это решение явно разделяет политику хранения для медиа и для проектных артефактов.

## 2026-05-07 - Гигиена Корпуса RAG И Исключение Дублей

Решение: `scripts/01_inventory.py` поддерживает `exclude_path_patterns` и исключает из индексации архивные, черновые и служебные копии: `Архив`, `Черновики и шаблоны`, backup-папки и файлы с `backup` в имени, `_analysis/docx_json*`, `_analysis/site_review*`, скрипты и JSON/Docx-слепки внутри `_analysis`. Curated markdown-аналитика в `_analysis/*.md` остается допустимым источником.

Почему:

- рабочая проектная папка содержит несколько версий одних и тех же документов;
- JSON-слепки `_analysis/docx_json*` дублируют исходные `.docx` и забивают top-k;
- архивы и черновики полезны как файловая история, но не должны быть источниками для продуктивного RAG;
- часть аналитических `.md` в `_analysis` содержит ценные выводы по замечаниям и расхождениям, поэтому исключать всю `_analysis` слишком грубо.

Правило сопровождения: exclude-паттерны добавляются итеративно и проверяются контрольными запросами. Запрещено заменять точные правила широким исключением вроде `_analysis/**`, пока не доказано, что внутри нет полезных curated-источников.

## 2026-05-07 - Query Дедуплицирует Одинаковые Chunks

Решение: `scripts/04_query.py` и numpy backend по умолчанию фильтруют пути из `exclude_path_patterns` и дедуплицируют одинаковые chunks по тексту. Для диагностики доступны `--include-excluded` и `--no-dedupe`.

Почему:

- даже после чистки корпуса могут оставаться одинаковые HTML/TXT-экспорты с разными путями;
- пользовательский top-k должен показывать разные смысловые источники, а не несколько копий одного фрагмента;
- флаги диагностики позволяют временно увидеть исходное поведение без изменения индекса.

## 2026-05-07 - `meeting_id` Является Slug, А Не UUID

Решение: папка встречи и поле `meeting_id` строятся как человекочитаемый slug `YYYY-MM-DD__short-title`, например `2026-05-07__project-sync`.

Почему:

- карточки встреч будут часто открываться вручную в проводнике и Git diff, поэтому имя должно быть читаемым;
- дата в идентификаторе сразу показывает порядок встреч без обращения к базе;
- MVP однопользовательский, поэтому глобальная уникальность UUID сейчас не нужна;
- если в один день появятся две встречи с одинаковой темой, конфликт решается суффиксом slug, например `project-sync-2`.

## 2026-05-07 - `meeting.json` Становится Машинным Контрактом Карточки Встречи

Решение: формальная схема карточки встречи хранится в `configs/schemas/meeting.schema.json`, а человекочитаемый шаблон - в `docs/templates/MEETING_CARD.md`.

Почему:

- будущие watcher, offline-процессинг, live-сессия, UI и API должны писать один и тот же формат;
- JSON schema позволяет проверять карточку без запуска полного pipeline;
- markdown-шаблон нужен для ручной работы и объясняет, как читать папку встречи без кода.

## 2026-05-07 - Retention И RAG-Политика Фиксируются В `meeting.json`

Решение: у каждой встречи в `meeting.json` есть блоки `retention` и `rag`. В RAG попадают структурированные артефакты и финальный transcript, а исходные медиа и временные live-файлы явно перечисляются как `no_index_artifacts`. Встречи с `retention.policy = protected` не подпадают под автоматическую ротацию медиа.

Почему:

- политика хранения и политика индексации должны быть видны на уровне конкретной встречи;
- protected-встречи часто содержат демонстрацию экрана или уникальные проектные решения;
- исходные видео и аудио полезны для восстановления, но не являются текстовым источником RAG;
- явные списки `indexed_artifacts` и `no_index_artifacts` упрощают будущий incremental update.

## 2026-05-07 - Версии `meeting.json` Меняются Только Через Явную Миграцию

Решение: `schema_version` в `meeting.json` фиксируется как `1` до появления осознанной несовместимой схемы. Новая версия схемы выпускается только вместе с описанием изменений и будущим upgrade-скриптом для старых карточек.

Почему:

- карточки встреч будут жить дольше, чем конкретная реализация pipeline;
- неявная миграция опасна для решений, задач и ссылок на источники;
- schema validation должна явно показывать, что карточка старого формата требует upgrade;
- diarization, расширенные участники или новые типы retention лучше вводить через версионирование, а не через неявные дополнительные поля.

## 2026-05-07 - Markdown-Карточка Является Представлением JSON-Артефактов

Решение: человекочитаемые таблицы в карточке встречи являются представлением `artifacts/decisions.json`, `artifacts/tasks.json`, `artifacts/risks.json` и других структурированных артефактов. Источником истины остаются JSON-файлы.

Почему:

- иначе решения, задачи и риски быстро разойдутся между Markdown и JSON;
- будущий UI/API должен читать структурированные артефакты, а не парсить Markdown;
- Markdown нужен для ревью человеком, но не должен становиться второй базой данных.

## 2026-05-08 - Артефакты Встречи Имеют Отдельные JSON-Схемы

Решение: решения, задачи, риски и открытые вопросы встречи хранятся отдельными JSON-файлами в `artifacts/` и валидируются отдельными схемами: `meeting.decisions.schema.json`, `meeting.tasks.schema.json`, `meeting.risks.schema.json`, `meeting.open_questions.schema.json`.

Почему:

- разные типы артефактов живут по разным правилам и не должны смешиваться в одном рыхлом JSON;
- каждый item должен иметь `source_refs`, чтобы memo/protocol можно было проверять по transcript segment или RAG-источнику;
- отдельные файлы проще переиспользовать в UI, API, RAG-индексации и будущем DOCX-экспорте;
- `needs_review` фиксирует сомнительные выводы модели без удаления полезного кандидата из результата.

Следствие: Markdown memo и протокол являются удобным представлением, но машинный источник истины для решений, задач, рисков и вопросов - соответствующие JSON-файлы.

## 2026-05-08 - Extractive-Режим Для Первого `summarized`

Решение: первый генератор итогов встречи `scripts/07_generate_meeting_artifacts.py` по умолчанию работает в `extractive`-режиме только как скаффолд контракта, а не как продуктовый генератор memo/protocol.

Почему:

- на текущем CPU-профиле Qwen3 может не успевать обработать длинный transcript за разумный timeout;
- для проверки контракта важнее надежно пройти путь `transcribed -> summarized`, чем сразу получить идеальный протокол;
- extractive-режим создает проверяемые `source_refs` на transcript segments и помечает все выводы `needs_review = true`;
- качество extractive зависит от ключевых слов и может превращать вопросы в задачи, риски в решения и сырые речевые обрывки в артефакты;
- продуктовый путь FTT-MA-12 должен строиться через LLM map-reduce-render или ручную редактуру поверх структурированных JSON.

## 2026-05-08 - FTT-MA-12 Строится Через Map-Reduce-Render

Решение: production-путь генерации итогов встречи строится как `MAP -> REDUCE -> RENDER`: MAP извлекает элементы из окон transcript по 6 минут с overlap 30 секунд, REDUCE объединяет и дедуплицирует JSON-артефакты, RENDER строит `memo.md` и `protocol.md` из финальных JSON. Один большой prompt на весь transcript и три независимых запроса `memo/protocol/json` не считаются production-путем.

Почему:

- короткие окна стабильнее для CPU-моделей и не требуют длинного контекста;
- частичные JSON можно сохранять и перезапускать только упавший этап;
- `memo.md` и `protocol.md` не расходятся с `decisions.json`, `tasks.json`, `risks.json` и `open_questions.json`;
- строгая классификация не позволяет превращать вопросы в задачи, риски в решения и отрицания действий в поручения.

## 2026-05-08 - WhisperX Отложен До Word-Level Timeline Или Diarization

Решение: WhisperX отложен до появления потребности в word-level timestamps или diarization. Текстовое качество на тестовой встрече не превосходит `large-v3-turbo`, время выполнения хуже. См. `docs/references/WHISPERX_EXPERIMENT.md`.

## 2026-05-08 - Внешнее Исследование Подтвердило Текущую Архитектуру Встреч

Решение: внешний аналитический документ `deep-research-report (1).md` из Downloads используется как подтверждающий research-вход, но не как прямой mandate на расширение архитектуры. Текущими подтвержденными решениями остаются:

- `MAP -> REDUCE -> RENDER` вместо full-context обработки длинной встречи;
- `faster-whisper` как основной ASR для CPU/local-first профиля;
- `large-v3-turbo/int8` как качественный offline-профиль для важных встреч, а `small/int8` как быстрый черновик/live MVP;
- `pyannote 3.1` как направление diarization, когда появится конкретная потребность;
- карточка встречи как папка с `meeting.json`, transcript и artifacts;
- обязательные `source_refs` во всех структурированных артефактах встречи.

Что не берем сейчас:

- мультиагентные оркестраторы вроде LangGraph/CrewAI: это преждевременная абстракция для текущего MVP;
- plain Whisper `large-v3` без turbo: слишком дорогой CPU-путь относительно `large-v3-turbo`;
- DeepEval/RAGAS как обязательный evaluation framework: для пет-проекта достаточно ручного baseline-набора и JSON/raw логов;
- cloud-сервисы Otter, Fathom, AssemblyAI и аналоги: они конфликтуют с local-first режимом;
- маркетинговые метрики вроде "50-70% time savings": не использовать как критерии приемки без собственного воспроизводимого замера.

Backlog-направления остаются точечными: `source_type`, hybrid retrieval, pyannote diarization, DOCX export и числовые метрики hallucination/source-grounding добавляются только под конкретную потребность или regression-контроль.

## 2026-05-12 - Project-Only Chatbot Как Первый Узкий UI

Решение: перед большим локальным UI сделать отдельный `Project-Only Chatbot MVP`. Это очищенный чат-бот поверх текущего numpy-RAG, который отвечает только по проектным источникам и отказывает, если вопрос не связан с проектом или не подтверждается найденными chunks.

Почему:

- RAG уже собран и пригоден для первых проектных вопросов, поэтому чат даст быструю пользовательскую ценность;
- большой UI для inbox, встреч, jobs и документов пока преждевременен;
- чат-бот проверит будущий контракт `FTT-MA-16` локального API на самом полезном сценарии: `chat -> search -> answer with sources`;
- project-only guardrail нужен до подключения OpenWebUI или любого другого интерфейса, чтобы модель не отвечала из общих знаний;
- отказ без источников лучше, чем правдоподобный ответ без трассировки.

Следствие: для MVP чат-бота обязательны `sources`, `score`, `chunk/file` и явный `refusal_reason` при отказе. Ответ без источников считается ошибкой продукта.

## 2026-05-12 - Оконный Offline-Pipeline Для Записей

Решение: для готовых записей на диске вводится отдельный `scripts/08_process_meeting_pipeline.py`. Он режет медиа на окна, транскрибирует каждое окно, сразу запускает MAP для готового окна, затем выполняет REDUCE и RENDER по валидным partial JSON.

Почему:

- полная схема "сначала вся ASR, потом вся LLM-обработка" плохо использует время ожидания;
- partial JSON нужно сохранять на диск, чтобы после падения REDUCE/RENDER не пересчитывать ASR и MAP;
- Ollama и Whisper конкурируют за CPU, поэтому параллелизм по умолчанию ограничен `1 ASR` и `1 LLM`;
- `source_refs` от модели нельзя принимать буквально: финальные ссылки нормализуются по реальному `transcript/segments.jsonl`, чтобы `segment_index`, `start` и `end` существовали и были абсолютными таймкодами встречи;
- `artifacts/pipeline_report.md` нужен как короткий операционный след прогона.

Статус: 2-window smoke на тестовой встрече прошел. Полный прогон всей встречи и новая запись `2026-05-08 14-32-43.mp4` остаются следующими задачами.

## 2026-05-12 - Быстрый профиль Project-Only Chatbot CLI

Решение: для интерактивного CLI-чата `scripts/09_chat.py` использовать быстрый профиль по умолчанию, а не полные значения RAG из `config.yaml`.

Параметры быстрого профиля:

- `top_k = 4`;
- `max_context_chars = 6000`;
- `source_char_limit = 1800`;
- `num_predict = 700`;
- `timeout_sec = 180`;
- prompt содержит `/no_think` и требование краткого ответа.

Почему:

- локальный smoke показал, что `--sources-only` быстро возвращает источники из `data/numpy_index`, значит задержка возникает не на retrieval;
- задержка возникает на генерации ответа локальной LLM;
- при `qwen3:8b`, `top_k=12` и `max_context_chars=24000` prompt становится слишком тяжелым для CPU и интерактивного режима;
- project-only чат должен быть первым быстрым пользовательским интерфейсом RAG, поэтому по умолчанию должен ограничивать контекст и длину генерации;
- для диагностики сохраняются ручные параметры CLI: `--top-k`, `--max-context-chars`, `--source-char-limit`, `--num-predict`, `--timeout-sec`, `--model`.

Следствие: полный/тяжелый профиль можно запускать вручную, но дефолтный CLI-профиль должен оставаться быстрым. Для быстрых проверок использовать `--sources-only --json`, а для полного ответа на CPU начинать с `--top-k 3 --max-context-chars 4500 --source-char-limit 1200 --num-predict 400 --timeout-sec 120`.

## 2026-05-27 - NTK Project Markers И Lexical-First Hybrid Для Точных Якорей

Решение: Project Knowledge Bot получил расширенные project-scope markers для NTK corpus: ФТТ/ПР/ЦТА/СоИ/ПМИ/Паспорт ИС, performance anchors (`2520`, `600 одновременно`), CTA anchors (`RTO`, `RPO`, `PostgreSQL`, `MinIO`, `Kubernetes`), AD/LDAPS/app_ccpm, MDR/НСИ/Bearer Token, НСИ-регламенты и экспорт PDF/Excel/CSV.

Для точных якорей hybrid retrieval временно смещает вес в сторону BM25/lexical (`0.58`) против vector (`0.42`). Это нужно, чтобы короткие точные термины вроде `app_ccpm`, `Bearer Token`, `2520`, `RTO/RPO`, `LDAPS` не терялись в embedding-поиске.

Почему:

- smoke по NTK corpus показал false-clarify на проектных вопросах без явного названия документа;
- `Паспорт ИС` с хвостом "связанные документы" ошибочно уходил в отказ;
- `карбонара` уходила в clarify, а должна быть out-of-scope/refused;
- для `app_ccpm` чистый BM25 находил `СоИ AD`, но hybrid с высоким vector-весом отдавал верхние места `ПР`/`ЦТА`.

Результат: `scripts/asu_june_bot_ntk_smoke_eval.py --mode hybrid` на 20 контрольных вопросах NTK corpus дал `20/20 ok`, `source_url_in_top5=19/20`, `failed_ids=[]`.

Следствие: дефолтный корпус всё ещё не переключается автоматически. Перед переключением нужен ручной просмотр ответов и источников по `data/asu_june_bot_ntk/smoke_eval_hybrid_after_markers_v3.jsonl`.
