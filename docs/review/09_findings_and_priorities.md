## 1. Как Читать

Каждая находка имеет фиксированную структуру:

- **Симптом** — что наблюдается извне.
- **Корень** — что в коде/доках вызывает симптом.
- **Риск** — что плохого может случиться, если не починить.
- **Цена починки** — XS / S / M / L (часы / 1 день / 2-3 дня / неделя+).
- **Зависимости** — что нужно сделать сначала.
- **Источник** — артефакт ревью, где разбирается подробно.

**Tier A** — рабочая ненадёжность сегодня. Что-то может молча сломаться.
**Tier B** — архитектурный долг. Не мешает сегодня, мешает завтра.
**Tier C** — гигиена. Косметика, путает читателя.

Это **engineering observations**, не предписания. Цена и приоритеты — для калибровки, не для принятия решений вместо владельца.

## 2. Tier A — Критично

### A1. Нет защиты от параллельного запуска 06/07/08 над одной встречей

**Симптом:** если случайно запустить два инстанса (например, `06` + `08` одновременно на одной папке), `meeting.json` будет переписан с race condition. Schema validation pre-mutate спасёт только от грубо невалидного состояния — не от логической потери transitions.

**Корень:** только `03_build_index.py` имеет `FileLock` (`logs/build_index.lock`). 06/07/08 валидируют `processing_status` через `ensure_status_allows_run`, но между чтением и записью **нет lock'а**. Это TOCTOU.

**Риск:** потеря `transcript/segments.jsonl` или `artifacts/*.json` при двойном запуске. Вероятность — низкая (пользователь сейчас один), но при добавлении watcher / pipeline-orchestrator вырастет.

**Цена:** S. Один `FileLock` per meeting (`meetings/<id>/.processing.lock`) с переиспользованием `rag_common.FileLock`.

**Зависимости:** нет.

**Источник:** артефакт №5 раздел 5.

### A2. Нет проверки уникальности `meeting_id`

**Симптом:** две встречи в один день с одинаковым slug → второй прогон молча перезапишет папку первой.

**Корень:** `06_transcribe_meeting.py:resolve_context` принимает `--meeting-dir` напрямую. Если пользователь скопировал шаблон и не поменял slug, никто не проверяет. `decisions.md 2026-05-07` декларирует «конфликт решается суффиксом `project-sync-2`», но enforcement в коде нет.

**Риск:** silent data loss. Особенно при автоматическом создании карточек встреч из watcher (когда watcher появится).

**Цена:** XS. Pre-check «директория не пустая → ругаться» в 06/08 + либо warning, либо auto-suffix.

**Зависимости:** нет.

**Источник:** артефакт №4 раздел 6.

### A3. Два sensitive-фильтра с разными паттернами

**Симптом:** запрос `"какие developer message в .env"` через `04_query.py --raw` — **запишется в query_log.jsonl**. Тот же запрос через `09_chat.py` — отказ с `refusal_reason=sensitive_or_system_request` и **в лог не попадёт**.

**Корень:** `rag_common.SENSITIVE_QUERY_PATTERNS` (12 паттернов) — узкий фильтр для записи в лог. `09_chat.SENSITIVE_PATTERNS` (20+ паттернов) — широкий фильтр для отказа в ответе. Они **не синхронизированы**.

**Риск:** sensitive-запрос (включая попытку извлечь промпт системы) может оказаться в `data/query_log.jsonl`, который не коммитится, но физически лежит на диске. Privacy-issue малой серьёзности.

**Цена:** XS. Унифицировать `SENSITIVE_PATTERNS` в `rag_common` (брать самый широкий список), оставить опциональный narrower фильтр для отказа в чате.

**Зависимости:** нет.

**Источник:** артефакт №3 раздел 4.1, артефакт №6 раздел 3.2.

### A4. `embeddings_cache.jsonl` накапливает stale-записи бесконечно

**Симптом:** после нескольких чисток корпуса в cache остались записи для chunks, которые больше не существуют. `03_build_index.py:128-129` это **явно фиксирует** в stdout (`stale_cached_embeddings`), но **никогда не очищает**.

**Корень:** append-only стратегия из `decisions.md 2026-05-06 Сохранять Embedding Cache`. Mode сознательный — для resumability. Но без compact-операции файл растёт навсегда.

**Риск:** через 6-12 месяцев `embeddings_cache.jsonl` будет, например, 500 MB вместо реальных 50 MB. Замедлит загрузку индекса, увеличит backup-объём.

**Цена:** S. Опциональный `--compact` для `03_build_index.py`, который пересобирает cache, оставляя только записи для текущих chunk_id. Или отдельный `compact_embedding_cache.py`. Должен быть гарантированно атомарным (через `.tmp + os.replace`).

**Зависимости:** A1 косвенно (нет смысла compact'ить, если concurrent build пишет append).

**Источник:** артефакт №3 раздел 3.5, артефакт №8 раздел 5.

### A5. Конфиги, которые не читаются — путают новых разработчиков

**Симптом:** в `config.example.yaml` есть целые секции (`transcription:`, `live_transcription:`, `rag.search_backend`), которые **никем не читаются**. Если пользователь поменяет `transcription.model: "large-v3"` в надежде, что 06 подхватит — ничего не изменится. 06 берёт модель из CLI `--model small`.

**Корень:** конфиг рос быстрее кода. Контракты под будущее остались как `placeholder` без пометки.

**Риск:** silent поведение — пользователь не понимает, почему его настройка не применяется.

**Цена:** XS-S. Либо удалить мёртвые ключи из example, либо реально их читать (тогда S — нужно пробросить в `06`/`08`). Минимальный fix: добавить комментарий `# Зарезервировано, в коде сейчас не читается` к каждому dead-ключу.

**Зависимости:** нет.

**Источник:** артефакт №6 раздел 2.

## 3. Tier B — Важно

### B1. asu_june_bot v2 корпус не имеет потребителя

**Симптом:** `data/asu_june_bot/chunks_v2.jsonl` накапливается через `extract_text_v2 → build_chunks_v2`, но **никто** этот файл не читает. Hybrid retriever подпроекта работает над **основным** `data/chunks.jsonl`.

**Корень:** план был «v1 (поверх старого корпуса) → v2 (полная замена)». v2 extract и chunk сделаны; `index_v2` упомянут как `future:`, но не реализован. Между v1 и v2 нет моста.

**Риск:** **40+ часов работы лежит без применения**. Богатые метаданные (`requirement_id`, `scenario_id`, `module`, `stage` для каждого chunk) не используются. Каждая новая чистка/перезапуск v2 — впустую.

**Цена:** M. Либо реализовать `scripts/asu_june_bot_build_index_v2.py` (numpy/FAISS поверх chunks_v2 + parent/block aware retrieval), либо явно заморозить v2 и удалить артефакты до решения.

**Зависимости:** требуется решение пользователя: «продолжаем v2 или замораживаем».

**Источник:** артефакт №7 раздел 1.

### B2. `decisions.md` декларирует функциональность, которой нет

**Симптом:** 4 решения зафиксированы как «принятые», но кода нет:

| Решение | Что декларировано |
|---|---|
| Live-транскрибация хранит MIC/SYS | Live pipeline не существует |
| В RAG индексируются артефакты встречи | Incremental indexer не существует |
| Markdown = представление JSON | Render отдельный LLM, без enforce'а |
| Retention `protected` ротация медиа | Ротация не реализована |

**Корень:** decisions.md как **исторический журнал решений** работает корректно. Но без статуса реализации он читается как «всё это уже сделано».

**Риск:** новый контрибьютор будет искать в коде то, что задекларировано, и тратить время впустую. AI-агент в новой сессии может принимать decisions.md как state-of-truth.

**Цена:** XS. Добавить колонку `Status` (`implemented` / `partial` / `planned` / `superseded`) к каждой записи decisions.md. Или отдельный `docs/decisions_status.md` с этой таблицей.

**Зависимости:** нет.

**Источник:** артефакт №6 раздел 8, артефакт №8 раздел 5.1.

### B3. Schema-резерв в `meeting.json` без маркировки «planned»

**Симптом:** `meeting.schema.json` содержит секции `classification`, `links`, `rag.indexed_artifacts`, `processing_status=classified|indexed`, `artifacts.classification_report`. Все optional, **никем не пишутся**.

**Корень:** схема была написана сразу с прицелом на полную целевую функциональность. Это **архитектурно правильно** для будущего, но смешивает «текущий контракт» с «roadmap».

**Риск:** косвенный — потребитель схемы (например, будущий UI) может рассчитывать на эти поля.

**Цена:** XS-S. Один из вариантов:
- (a) добавить `description: "Reserved for FTT-MA-13 classifier"` в каждое reserved-поле;
- (b) разделить на `meeting.schema.json` (current) и `meeting.future.schema.json` (full).

(a) предпочтительнее — даёт читателю явный маркер без дублирования.

**Зависимости:** нет.

**Источник:** артефакт №3 раздел 5.1, артефакт №6 раздел 4.

### B4. `decision_id` / `task_id` / `risk_id` только локально уникальны

**Симптом:** `DEC-001` в встрече A ≠ `DEC-001` в встрече B. При попытке кросс-меетинговых ссылок (`links.related_decisions[]`) нужен составной ключ, которого нет.

**Корень:** ID назначаются на REDUCE per-meeting через увеличивающийся счётчик. Глобальная уникальность не enforced.

**Риск:** блокирует FTT для cross-meeting linking. На сегодня — не активно, но при попытке реализовать `meeting.json::links.related_decisions[]` придётся менять формат ID.

**Цена:** S, **если делать сейчас**. Один из вариантов:
- (a) сменить формат на `<meeting_id>:DEC-001`;
- (b) сменить на UUID;
- (c) оставить, но enforce'ить составной ключ `(meeting_id, decision_id)` на уровне линковок.

(c) совместим с текущими файлами.

**Зависимости:** A2 (если меняем формат ID — заодно стоит решить с meeting_id collision).

**Источник:** артефакт №4 раздел 6.

### B5. Два параллельных чата: `09_chat` и `asu_june_bot_search`

**Симптом:** в репо живут два чат-CLI с разными моделями (`qwen3:8b` vs планируемый `qwen3:4b`), разными retrieval-стратегиями (numpy-only vs hybrid), разными prompts, разными sensitive-фильтрами.

**Корень:** `09_chat.py` объявлен «prototype» в `docs/subprojects/asu-june-bot/todo.md`, но remains in production. `asu_june_bot_search.py` — только поиск, чат-CLI не существует.

**Риск:** долг растёт. Каждое улучшение в `09_chat` (например, новый prompt, новая модель) нужно повторять в подпроекте, когда тот дойдёт до чата.

**Цена:** L, если действительно мигрировать. Либо
- (a) `09_chat` остаётся, hybrid retriever подключается опцией;
- (b) построить `asu_june_bot_chat.py` отдельно, оставить `09_chat` как deprecated;
- (c) консолидировать в один CLI.

**Зависимости:** B1 (нужно сначала решить судьбу v2).

**Источник:** артефакт №7 раздел 6.

### B6. Дублирующиеся утилиты в `rag_common` и `src/asu_june_bot/ingestion/utils`

**Симптом:** 7 функций имеют две независимые реализации (`normalize_text`, `stable_id`, `sha256_file`, `read_text_guess`, `jsonl_write`, `load_config`, `resolve_work_path`). `stable_id` уже дрифтит (`length=24` в main, `length=32` в подпроекте).

**Корень:** сознательное архитектурное разделение namespace ради независимости подпроекта.

**Риск:** дрифт растёт. Если в `normalize_text` основной кодабазы будет добавлено правило (например, удаление zero-width characters), оно не попадёт в v2 extractor.

**Цена:** S. Либо
- (a) вынести в `src/_shared/` пакет, оба импортируют оттуда;
- (b) подпроект импортирует через `from scripts.rag_common import ...` (hack через sys.path, как сейчас в `vector.py`);
- (c) оставить дублирование, явно задокументировать в обоих файлах со ссылкой друг на друга.

(a) — правильнее long-term, но требует пакетной реструктуризации.

**Зависимости:** связано с B5.

**Источник:** артефакт №7 раздел 2.3, 5.4.

### B7. Тестов нет вообще

**Симптом:** `tests/` отсутствует. Ни одного `test_*.py`. Проверки качества — ручные через baseline-Markdown.

**Корень:** проект развивался быстро, тестовая инфраструктура отложена.

**Цена:** M для базового набора (pytest + 30 тестов). L для полного coverage.

**Зависимости:** нет.

## 5. Что Уже Работает Хорошо

Эту секцию ввожу специально, чтобы ревью не выглядело как «всё плохо». Несколько архитектурных решений реализованы **очень аккуратно**:

### 5.1 Schema-driven state machine для встреч

`06`/`07`/`08` валидируют `meeting.json` через `jsonschema Draft 2020-12` **перед каждой мутацией**. `ensure_status_allows_run` гарантирует, что только разрешённые transitions проходят без `--force`. Это **золотой стандарт** того, как должны выглядеть контракты в репозитории. Стоит расширять этот паттерн на другие сущности (config, query_log, etc.).

### 5.2 Атомарная сборка numpy index

`rag_numpy_backend.build_index` пишет в `index_dir.tmp/`, потом `shutil.move` в финальный путь. Гарантирует, что неполный индекс не виден потребителям. Pattern переиспользуется через `write_json_atomic` в `07/08`.

### 5.3 `decisions.md` как формат

24 датированных решения с шаблоном «Решение: X. Почему: bullet list». Никогда не удаляется; уточнения добавляются датированными подзаписями («Уточнение от 2026-05-07»). Это **идеальный формат архитектурного журнала**. Его недостаток — отсутствие статуса реализации (см. B2) — легко чинится.

### 5.4 `AGENTS.md` компактен и плотен
