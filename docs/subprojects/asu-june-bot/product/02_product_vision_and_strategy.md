# 02. Видение Продукта и Стратегия

Обновлено: 2026-05-15.

## Видение

`Asu June Bot` — локальный project-only AI-ассистент, который превращает документацию и артефакты проекта ЦП УПКС в рабочую аналитическую память: с ответами по источникам, междокументными связями и контролируемыми отказами вне области проекта.

## Продуктовая Формула

Не "универсальный чат с файлами", а:

```text
проектная память
+ дисциплина предметной области
+ проверяемые источники
+ локальный контур
= рабочий аналитический продукт
```

## Стратегическая Ставка

Главная ставка продукта — глубина в одной проектной области, а не широта.

Мы не пытаемся сделать ботом "про все". Мы делаем инструмент, который:

- знает конкретный проектный корпус;
- умеет работать с конкретными типами артефактов;
- поддерживает аналитическую, а не только поисковую работу;
- ценен тем, что не выходит за рамки своей области.

Это ближе к идее `Zero to One`: создать специфическую ценность, которую трудно получить из обычного generic AI-чата.

## Какую Позицию Занимает Продукт

Продукт занимает позицию:

```text
не поисковик по папке
не "умный чат обо всем"
не генератор красивых, но неподтвержденных ответов

а локальный ассистент системного аналитика по конкретному проекту
```

## Базовые Принципы Продукта

### 1. Scope Discipline

Лучше корректно отказаться, чем ответить красиво, но вне корпуса проекта.

### 2. Source-Backed Answering

Каждый полезный ответ должен быть подтверждаем проектными источниками.

### 3. Cross-Document Reasoning

Ценность не только в том, чтобы найти один chunk, а в том, чтобы показать связь между:

- ФТТ;
- ЦТА;
- ПМИ;
- проектными решениями;
- протоколами встреч;
- аналитическими артефактами.

### 4. Local-First Execution

Продукт должен работать локально и не зависеть критически от внешней SaaS-инфраструктуры.

### 5. Reproducibility

Корпус, индекс, правила поиска, guard-логика и документация должны быть воспроизводимы через Git и локальные runbook.

## Какой Продукт Мы Строим На Первом Этапе

На первом этапе это не "полноценный корпоративный AI-помощник", а:

- надежный search core;
- project-only filter/guard;
- API `/search`;
- затем API `/chat`;
- затем UI-оболочка;
- затем более глубокая аналитика и связка со встречами.

## Продуктовая Гипотеза

Если пользователю дать локального project-only ассистента, который:

- отвечает по корпусу проекта;
- не смешивает проектные данные с общими знаниями модели;
- показывает источники и ограничения;
- стабильно работает на рабочем корпусе документов,

то пользователь будет использовать его как постоянный рабочий инструмент для анализа проектной документации, а не как разовый эксперимент.

## Что Будет Ядром Ценности

Ядро ценности `Asu June Bot`:

1. надежный поиск по проекту;
2. защита от ложного scope;
3. структурированный ответ с источниками;
4. удержание проектного контекста между документами и встречами;
5. путь от поиска к аналитическим выводам.

## Что Не Должно Размывать Стратегию

Стратегию нельзя размывать следующими соблазнами:

- превращать продукт в общий чат "про все";
- рано уходить в UI раньше стабильного search/chat ядра;
- смешивать продуктовую память и шумные system exports;
- подменять качество ответов большим количеством LLM-магии;
- строить сложную агентную систему до появления стабильной базовой пользы.

## Формула Успеха На Этапе MVP

MVP успешен, если пользователь говорит не "интересно, что умеет бот", а:

```text
теперь я реально быстрее нахожу и сверяю информацию по проекту
```
