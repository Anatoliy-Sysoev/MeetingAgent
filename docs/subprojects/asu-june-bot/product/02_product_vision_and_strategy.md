# 02. Видение продукта и стратегия

Обновлено: 2026-05-16.

## Видение

Project Knowledge Bot — локальный project-only AI-ассистент, который превращает документацию и артефакты проекта информационной системы в рабочую аналитическую память: с ответами по источникам, междокументными связями, контролируемыми отказами вне области корпуса и измеримым качеством ответов.

## Продуктовая формула

Не универсальный чат с файлами, а:

```text
проектная память
+ дисциплина предметной области
+ проверяемые источники
+ локальный контур
+ baseline качества
= рабочий аналитический продукт
```

## Стратегическая ставка

Главная ставка продукта — глубина в одной проектной области, а не широта.

Мы не пытаемся сделать бота "про все". Мы делаем инструмент, который:

- знает конкретный проектный корпус;
- умеет работать с типами проектных артефактов;
- поддерживает аналитическую, а не только поисковую работу;
- ценен тем, что не выходит за рамки своей области;
- измеряет качество через regression/eval cases.

## Позиционирование продукта

Продукт занимает позицию:

```text
не поисковик по папке
не универсальный чат обо всем
не генератор красивых, но неподтвержденных ответов

а локальный ассистент проектной команды по документации информационной системы
```

## Базовые принципы продукта

### 1. Scope discipline

Лучше корректно отказаться, чем ответить красиво, но вне корпуса.

### 2. Source-backed answering

Каждый полезный ответ должен быть подтверждаем проектными источниками.

### 3. Cross-document reasoning

Ценность не только в том, чтобы найти один chunk. Ценность в том, чтобы показать связь между требованиями, архитектурой, проектными решениями, интеграциями, испытаниями, протоколами и открытыми вопросами.

### 4. Local-first execution

Продукт должен работать локально и не зависеть критически от внешней SaaS-инфраструктуры.

### 5. Reproducibility

Корпус, индекс, правила поиска, guard-логика и документация должны быть воспроизводимы через Git и локальный runbook.

### 6. Quality before UI

Пользовательский интерфейс не должен появляться раньше baseline оценки `/chat`. Иначе красивый UI замаскирует слабое качество источников и ответов.

## Текущий продуктовый этап

Текущий этап — не корпоративная эксплуатация и не полноценный web-продукт.

Текущий этап:

```text
локальный Search API MVP
+ локальный Chat API MVP
+ QH-1 observability/eval baseline
```

Реализовано:

```text
GET /health
POST /search
POST /chat
ChatRunsLogger
Eval baseline runner
```

Открыто:

```text
baseline-прогон
source quality filter
parent expansion при необходимости
UI после стабилизации качества
```

## Продуктовая гипотеза

Если пользователю дать локального project-only ассистента, который:

- отвечает по корпусу проекта;
- не смешивает проектные данные с общими знаниями модели;
- показывает источники и ограничения;
- стабильно работает на рабочем корпусе документов;
- позволяет проверить качество на baseline cases,

то пользователь будет использовать его как постоянный рабочий инструмент для анализа проектной документации, а не как разовый эксперимент.

## Ядро ценности

Ядро ценности Project Knowledge Bot:

1. надежный поиск по проектной документации;
2. защита от ложного scope;
3. структурированный ответ с источниками;
4. удержание проектного контекста между документами и встречами;
5. переход от поиска к аналитическим выводам;
6. воспроизводимый baseline качества.

## Что не должно размывать стратегию

Стратегию нельзя размывать следующими соблазнами:

- превращать продукт в общий чат;
- рано уходить в UI раньше стабильного search/chat ядра;
- смешивать проектную память и шумные system exports;
- подменять качество ответов большим количеством LLM-магии;
- строить сложную агентную систему до появления стабильной базовой пользы;
- внедрять DSPy, LLM-as-judge или NLI как runtime dependency до накопления dataset.

## Формула успеха MVP

MVP успешен, если пользователь говорит:

```text
теперь я реально быстрее нахожу и сверяю информацию по проектной документации
```

а технически это подтверждается:

```text
/search стабилен
/chat стабилен
refused/clarify не вызывают LLM
answers имеют sources/citations
baseline eval формируется
качество можно улучшать измеримо
```
