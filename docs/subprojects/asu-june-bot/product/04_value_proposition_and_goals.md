# 04. Ценностное Предложение и Цели

Обновлено: 2026-05-15.

## Ценностное Предложение

`Asu June Bot` помогает системному аналитику и смежным ролям быстрее получать подтвержденные ответы по проекту ЦП УПКС, не смешивая проектные факты с общими знаниями модели.

Короткая формулировка:

```text
Найти, проверить, связать и объяснить проектные знания быстрее, чем вручную.
```

## Какие Выгоды Дает Продукт

### Функциональные выгоды

- быстрый поиск по проектной документации;
- ссылки на подтверждающие источники;
- восстановление контекста темы;
- связка нескольких документов в одном ответе;
- отказ от ответа, если данных нет или вопрос вне проекта.

### Когнитивные выгоды

- меньше потерь контекста;
- меньше ручной сверки;
- меньше риска "кажется, это было где-то там";
- выше уверенность в рабочем ответе.

### Организационные выгоды

- проще передавать контекст внутри команды;
- меньше зависимость от памяти конкретного человека;
- проще возвращаться к обсуждению через время;
- появляется единая project-only knowledge surface.

## Ценность Через Призму Value Proposition

### Customer Jobs

- найти ответ по проекту;
- доказать ответ источником;
- сверить документы между собой;
- восстановить логику решения;
- быстро войти в тему после паузы.

### Pains

- долго искать;
- легко потерять связь между документами;
- сложно быстро доказать вывод;
- риск опереться на устную память, а не на источник;
- generic AI отвечает уверенно, но непроверяемо.

### Pain Relievers

- project-only guard;
- retrieval по реальному корпусу;
- source-backed answer policy;
- контекст primary/supporting sources;
- честные ограничения и отказы.

### Gain Creators

- быстрый рабочий ответ;
- прозрачность источников;
- повторное использование проектной памяти;
- база для дальнейших сценариев: chat, traceability, meeting integration.

## Продуктовые Цели

### Цель 1. Надежный поиск по проекту

Пользователь должен стабильно находить релевантные источники по ключевым проектным вопросам.

### Цель 2. Контролируемая область применения

Продукт не должен притворяться общим помощником. Его сила — в ограничении области.

### Цель 3. Проверяемый ответ

Каждый полезный ответ должен быть поддержан понятными источниками.

### Цель 4. Основа для аналитических сценариев

Поиск должен стать фундаментом для следующего слоя: chat, cross-document analysis, meeting memory.

## KPI И Рабочие Метрики MVP

### Качество Guard

- `false_allow = 0` для критических regression cases;
- 100% блокировка очевидных out-of-project и mixed-scope запросов в baseline-наборе.

### Качество Search

- не менее 80% релевантных попаданий в top-5 по рабочему eval-набору;
- не менее 70% вопросов, где primary/supporting context позволяет собрать полезный аналитический ответ.

### Качество Ответа

- 0 ответов без project sources в production policy;
- 100% отказов или ограничений при отсутствии подтверждения.

### Полезность Для Пользователя

- пользователь тратит меньше времени на первичный поиск;
- продукт используется как первая точка входа в тему, а не как "последняя надежда".

## Антиметрики

Продукт считается неудачным, если:

- он часто отвечает вне области проекта;
- primary sources систематически нерелевантны;
- пользователь все равно сразу идет искать вручную;
- бот выдает красиво звучащий, но недоказуемый текст;
- продукт невозможно воспроизвести локально и поддерживать через Git/runbook.

## Формулировка Успеха

Продукт успешен, если для рабочей роли системного аналитика он становится не "демо AI", а привычным инструментом для:

- поиска;
- проверки;
- сопоставления;
- быстрого восстановления проектного контекста.
