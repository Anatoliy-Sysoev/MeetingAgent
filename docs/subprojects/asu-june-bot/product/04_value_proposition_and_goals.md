# 04. Ценностное предложение и цели

Обновлено: 2026-05-16.

## Ценностное предложение

Project Knowledge Bot помогает системному аналитику и смежным ролям быстрее получать подтвержденные ответы по проектной документации информационной системы, не смешивая факты из источников с общими знаниями модели.

Короткая формулировка:

```text
найти, проверить, связать и объяснить проектные знания быстрее, чем вручную
```

## Какие выгоды дает продукт

### Функциональные выгоды

- быстрый поиск по проектной документации;
- ссылки на подтверждающие источники;
- восстановление контекста темы;
- связка нескольких документов в одном ответе;
- отказ от ответа, если данных нет или вопрос вне корпуса;
- диагностика guard/retrieval/context/LLM.

### Когнитивные выгоды

- меньше потерь контекста;
- меньше ручной сверки;
- меньше риска "кажется, это было где-то там";
- выше уверенность в рабочем ответе;
- меньше зависимости от памяти отдельного человека.

### Организационные выгоды

- проще передавать контекст внутри команды;
- проще возвращаться к обсуждению через время;
- появляется единая project-only knowledge surface;
- ответы становятся воспроизводимыми;
- качество можно измерять через baseline.

## Ценность через Value Proposition

### Customer Jobs

- найти ответ по проекту;
- доказать ответ источником;
- сверить документы между собой;
- восстановить логику решения;
- быстро войти в тему после паузы;
- понять, когда данных недостаточно.

### Pains

- долго искать;
- легко потерять связь между документами;
- сложно быстро доказать вывод;
- риск опереться на устную память, а не на источник;
- generic AI отвечает уверенно, но непроверяемо;
- слабое качество retrieval трудно диагностировать без логов и eval.

### Pain Relievers

- project-only guard;
- retrieval по реальному корпусу;
- source-backed answer policy;
- primary/supporting/excluded context;
- честные ограничения, уточнения и отказы;
- chat_runs logging;
- baseline evaluation.

### Gain Creators

- быстрый рабочий ответ;
- прозрачность источников;
- повторное использование проектной памяти;
- база для traceability, meeting integration и analyst mode;
- измеримое улучшение качества через QH-этапы.

## Продуктовые цели

### Цель 1. Надежный поиск по проектной документации

Пользователь должен стабильно находить релевантные источники по ключевым проектным вопросам.

### Цель 2. Контролируемая область применения

Продукт не должен притворяться общим помощником. Его сила — в ограничении области и проверяемости.

### Цель 3. Проверяемый ответ

Каждый полезный ответ должен быть поддержан понятными источниками и citations.

### Цель 4. Основа для аналитических сценариев

Поиск и чат должны стать фундаментом для следующего слоя: traceability, cross-document analysis, meeting memory.

### Цель 5. Измеримое качество

Любое улучшение retrieval/context/LLM должно проверяться через eval baseline, а не только субъективным smoke.

## KPI и рабочие метрики MVP

### Качество Guard

```text
false_allow = 0 для критических regression cases
100% блокировка obvious out-of-project и mixed-scope в baseline-наборе
refused/clarify -> retrieval_called=false
```

### Качество Search

```text
>= 80% релевантных попаданий в top-5 по рабочему eval-набору
primary_sources содержат не шум, а usable evidence
```

### Качество Chat

```text
0 answered без sources
0 answered без citations
validation_errors=[] для smoke-approved ответов
llm_called=false при refused/clarify
```

### Наблюдаемость

```text
chat_runs.jsonl пишется
latency_ms фиксируется
eval reports создаются
ошибки распределяются по guard/retrieval/context/LLM/validator
```

### Полезность для пользователя

```text
пользователь тратит меньше времени на первичный поиск
бот используется как первая точка входа в тему
sources можно перенести в рабочий ответ или документ
```

## Антиметрики

Продукт считается неудачным, если:

- он часто отвечает вне области корпуса;
- primary sources систематически нерелевантны;
- пользователь все равно сразу идет искать вручную;
- бот выдает красиво звучащий, но недоказуемый текст;
- продукт невозможно воспроизвести локально и поддерживать через Git/runbook;
- UI появляется раньше измеримого качества ядра.

## Формулировка успеха

Продукт успешен, если для рабочей роли системного аналитика он становится не демонстрацией AI, а привычным инструментом для:

- поиска;
- проверки;
- сопоставления;
- быстрого восстановления проектного контекста;
- подготовки ответа с источниками.
