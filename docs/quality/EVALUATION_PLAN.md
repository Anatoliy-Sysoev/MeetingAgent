# План Оценки Качества

## Качество RAG

Текущий baseline RAG фиксируется в:

- `docs/quality/rag_eval_questions.md` - список вопросов и статусы;
- `docs/quality/rag_eval_report_template.md` - шаблон исторического отчета;
- `docs/quality/rag_eval_baseline_2026-05-07.md` - первый фактический baseline-прогон.

Нужно прогнать небольшой набор проектных вопросов:

- Какие документы нужны для сдачи?
- Что входит в Паспорт ИС?
- Какие разделы ФТТ влияют на архитектуру?
- Какие решения были приняты по интеграции?
- Какие документы упоминают приемку или передачу?

Для каждого вопроса проверять:

- релевантность ответа;
- релевантность источников;
- пропущенные источники;
- галлюцинации;
- полезность ссылок на источники.

Минимальный формат результата:

- вопрос;
- команда запуска;
- top источники;
- ожидаемые источники;
- оценка релевантности;
- замечания по дублям;
- решение: проходит / требует улучшения / провал.

Для компактного просмотра использовать:

```powershell
.\.venv\Scripts\python.exe scripts\04_query.py "Вопрос" --top-k 8 --compact
```

Для машинного журнала использовать:

```powershell
.\.venv\Scripts\python.exe scripts\04_query.py "Вопрос" --top-k 8 --raw
```

## Качество Встреч

Для каждой обработанной встречи проверять:

- точность транскрипта;
- полезность memo;
- структуру протокола;
- извлеченные решения;
- извлеченные задачи;
- качество классификации.

## Качество Генерации Документов

Для сгенерированных черновиков проверять:

- опору на источники;
- структуру;
- полноту;
- согласованность с проектной терминологией;
- объем ручной доработки.

## Минимальный План Тестов

Перед расширением продукта нужны тесты на критические участки:

- `chunk_text`: overlap, пустой текст, длинный текст, разрыв по абзацам;
- `rag_numpy_backend`: сборка индекса на синтетических данных и поиск ожидаемого источника;
- `04_query.py`: компактный и JSON-вывод без вызова LLM на синтетических contexts;
- extraction: хотя бы smoke-проверки DOCX, PDF, XLSX;
- будущий `meeting.json`: проверка обязательных полей и статусов.
